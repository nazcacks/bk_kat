# Phase Screen Designer — 개발설계 → EY 화면설계 HTML

BK 회계 플랫폼의 페이즈별 개발설계 문서(`D:\bk_kat\설계\개발설계\페이즈NN_*.md`)를, 이미 확정된 EY 레이아웃 화면설계서(`D:\bk_kat\설계\페이즈_화면\페이즈NN_화면.HTML`)와 **완전히 같은 시각 스타일**의 단일 자체완결 HTML로 변환한다. 외부 의존성 없음, 순수 바닐라 JS, 오프라인 열람 가능.

gold reference(스타일 기준)는 `페이즈01_화면.HTML`과 `페이즈02_화면_v1.0.html`이다. 이 둘이 룩앤필의 진실이며, 새 결과물은 이들과 나란히 놓아도 구분이 안 될 정도로 일치해야 한다.

## 핵심 원칙

화면설계서는 두 가지를 동시에 전달한다: (1) **목업(와이어프레임)** — 실제 화면처럼 보이는 그림으로, 대표 데이터를 채워 그 화면의 핵심 동작·규칙이 눈에 보이게 한다. (2) **설계 내용** — 목적/입력/기능/검증/산출물/연계를 구조화한 카드. 개발설계 md의 텍스트를 그대로 나열하는 게 아니라, "이 화면을 개발자가 보면 무엇을 만들어야 하는지 즉시 이해되는가"를 기준으로 목업과 카드에 배분한다.

CSS는 손대지 않는다. 색·간격·폰트·컴포넌트는 `assets/template.html`에 픽셀 단위로 맞춰져 있다. 새 스타일을 발명하지 말고 정의된 클래스만 조합한다.

## 워크플로

### 1단계 — 개발설계 소스를 완전히 읽는다
대상 md(예: `페이즈03_전표_자동전표.md`)를 전부 읽는다. 파일이 크면 offset/limit으로 나눠 끝까지 읽는다. 파악할 것:
- **화면 목록** — 보통 md의 §2 화면 인벤토리 표. 화면 ID(예: SA-JE-01), 명칭, 차수(1차/2차), 설계서 §근거, 메뉴 위치.
- **업무 흐름** — End-to-End, 경로/변형, 검증 게이트, 선행/후행 페이즈 관계.
- **화면별 소절** — 각 화면의 목적/입력 필드/기능·동작/검증/산출물/연계/권한.
- **횡단 내용** — 데이터 모델, API, 배치, 검증표, 권한·SOD, DoD 등. 이들은 화면이 아니라 공통 섹션으로 묶는다.

필요하면 상위 기준문서 `D:\bk_kat\설계\bk_설계서_v1.1.md`의 해당 §를 참조해 화면 의도를 보강한다.

### 1.5단계 — 참조 화면 HTML이 있으면 업무 내용과 스타일 기준을 분리한다
사용자가 `구현_화면_*.html`, `bk_화면설계서_v1.0.html`, 특정 `페이즈NN_화면_vX.Y.html`, 또는 `@파일명`으로 기존 화면을 지목하면, 그 파일의 **업무 내용·필드·검증·설명은 최우선 소스**다. 다만 시각 스타일은 대상 페이즈 화면의 기준을 따른다. 같은 EY 페이즈 스타일의 참조 화면은 원형 이식하고, 다른 자체 스타일 문서라면 업무 내용만 가져와 EY 페이즈 컴포넌트로 다시 그린다.

원칙:
- "같은 형태로 적용"은 대상 화면군의 기준 화면과 같은 `page-title → crumb/src → h2.sec → mock-cap → titlebar → qbar → actionbar → fnbar → grid → rpanel → statusbar → 상세 설명`의 **마크업과 시각 밀도까지 유지**한다는 뜻이다.
- 기준 HTML에서 대상 화면 섹션 경계를 찾는다. 예: `<section class="section visible" id="jv-input">`부터 다음 `<section ...>` 직전까지.
- 참조 HTML이 이미 EY 페이즈 스타일이면 대상 페이즈 파일에는 사이드바 연결을 위해 섹션 `id`만 바꾼다. 예: `jv-input` → `jnl01`. 필요하면 식별용 보조 클래스만 추가한다. 예: `<section class="section source-wjv" id="jnl01">`.
- 이 경우 내부 목업은 원본 섹션을 그대로 옮긴다. `titlebar`, `qbar`, `actionbar`, `fnbar`, `table.grid`, `rpanel`, `rp-box`, `statusbar`, 원본 설명 표/절을 임의로 단순화하거나 입력 컨트롤형 그리드로 바꾸지 않는다.
- 참조 HTML이 별도 문서 스타일(`wf-*`, 자체 nav, 자체 색상 변수 등)이면 그 스타일을 대상 페이즈로 가져오지 않는다. 업무 필드·검증·표 설명을 추출해 대상 기준 화면과 같은 EY 컴포넌트로 재작성한다.
- 원본 설명이 6카드보다 더 자세하면 삭제하지 않는다. `.screen-desc-list` 6카드는 페이즈 공통 설명이 필요할 때 추가할 수 있지만, 원본의 상세 설명 절(`화면 설명 ①`, `②`, `③`, 또는 1~15절)은 유지한다.
- 같은 EY 스타일 원본 섹션이 템플릿에 없는 보조 클래스를 사용하면 해당 클래스를 `assets/template.html`과 `references/components.md`에 반영한다. 자체 문서 스타일의 클래스는 반영하지 말고 EY 컴포넌트로 변환한다.
- 교체 후 원본 섹션에서 `id/class` 치환만 한 문자열과 대상 섹션이 같은지 비교한다. 다르면 의도한 변경(예: 섹션 id, source 보조 클래스, 페이즈 메뉴명)인지 설명할 수 있어야 한다.

실제 적용 사례:
- `D:\bk_kat\설계\화면설계\구현_화면_전표_v1.1.html`의 `WJV-01 일반전표` 섹션을 `D:\bk_kat\설계\페이즈_화면\페이즈03_화면_v1.0.html`의 `jnl01`에 적용할 때는 원본 섹션을 거의 그대로 사용하고, `<section class="section source-wjv" id="jnl01">`만 맞춘다. 상단 메뉴바 타이틀(`전표관리 › 일반전표`), 조회조건, 액션바, 스마트입력 기능바, 분개 그리드, 우측 패널, 하단 상태바, 상세 설명 ①~15절까지 유지한다.
- `D:\bk_kat\설계\화면설계\구현_매입매출_전표_v1.0.html`의 `WPV-01 매입매출전표입력`은 자체 와이어프레임(`wf-*`) 스타일이므로 그대로 이식하지 않는다. 업무 내용(상단 세무헤더, 하단 분개라인, taxType 11~25/51~64, 부가세 자동라인, 전자발행, 더보기, 검증/산출물)을 추출하고, `WJV-01 일반전표`와 같은 EY 고밀도 그리드 스타일(`<section class="section source-wjv" id="jnl02">`)로 재작성한다.

### 2단계 — 섹션 구조를 설계한다
전체 = `#home`(개요) + 화면별 섹션 N개 + 공통 섹션 몇 개. 사이드바 메뉴 그룹을 md의 업무 묶음대로 나눈다(예: "전기 재무제표", "초기이월", "2차 화면", "게이트·공통"). 각 화면에 짧은 소문자 섹션 id를 부여한다(예: SA-JE-01 → `je01`). 사이드바 `data-sec`와 섹션 `id`가 1:1로 일치하도록 미리 매핑을 정한다.

### 3단계 — 컴포넌트 사용법을 로드한다
`references/components.md`를 읽는다. 모든 섹션·부품의 copy-paste 스니펫과 클래스 규칙이 있다. 목업 부품 순서는 항상: `titlebar → qbar → actionbar → (tabbar) → mock-flex[lpane? · mock-main · rpanel?] → statusbar`. 화면 성격(입력/조회/설정/배치/대시보드)에 맞는 부품만 고른다.

### 4단계 — 템플릿으로 뼈대를 만든다
방식 A(앵커 삽입)면 `assets/template.html`을 Read로 읽어 결과 경로에 Write로 **그대로** 옮긴다(파일 시스템 `cp`에 의존하지 말고 Read→Write가 확실하다). 방식 B(조각파일)면 template를 head/sidebar/foot 3조각으로 나눠 build 폴더에 각각 Write한다. 어느 방식이든 플레이스홀더를 치환한다:
- `{{DOC_TITLE}}` — 예: `BK 페이즈03 화면설계 — 전표·자동전표 (SA-JE, EY 레이아웃)`
- `{{MENU_HEAD}}` — 예: `전표관리 · 자동전표 (P03)`
- `{{FOOTER_TEXT}}` — 예: `EY BK 회계 플랫폼 — 페이즈03 화면설계 v1.0 · 근거: 페이즈03_전표_자동전표.md · 레이아웃 준거: 페이즈01_화면.HTML`
- `{{SIDEBAR_NAV}}` — 2단계에서 정한 `.mgroup` 블록들.
- `{{SECTIONS}}` — 이 자리는 5단계에서 채운다. 지금은 `<!-- SECTIONS_ANCHOR -->` 주석 한 줄만 남겨 둔다. 이후 각 편집에서 이 앵커 **앞에** 섹션을 삽입(앵커를 `새 섹션들 + 앵커`로 치환)하면 순서대로 안전하게 이어붙일 수 있다.

이모지·기호는 gold reference처럼 **raw 글리프**(🔍 ☰ ★ 🏠 📅 ▸ ▾ ✓)를 그대로 쓴다. 파일이 UTF-8이므로 HTML 엔티티로 바꿀 필요 없다.

### 5단계 — 섹션을 채운다 (반복 · 반드시 여러 번에 나눠서)
결과물은 크다(수만 자). 출력 토큰 한계 때문에 한 번의 편집으로 전체를 쓸 수 없으니 **처음부터 6~10회 정도로 쪼개** 만들 계획을 세운다. 순서: 먼저 `#home`(개요: 업무흐름 flow + 화면 인벤토리 spec 표 + 선행조건 표) → 화면 섹션을 사이드바 그룹 단위로 → 마지막에 공통 섹션. 각 화면 섹션은 반드시 `.mock`(대표 데이터가 든 목업) + `.screen-desc-list`(6칸 카드)를 갖는다.

조립 방식은 두 가지다. 화면 수가 많거나(대략 15개 초과) 결과물이 128KB를 넘을 것으로 예상되면 **방식 B(조각파일 + cat 조립)를 권장**한다 — 페이즈05(32섹션·186KB)에서 잘림 없이 검증됨.

- **방식 A — 앵커 삽입(소·중 규모):** 4단계에서 `<!-- SECTIONS_ANCHOR -->` 한 줄만 남기고, 각 편집에서 앵커를 `새 섹션들 + 앵커`로 치환해 순서대로 이어붙인다. 마지막 편집에서 앵커를 제거한다. Edit 도구가 파일 끝을 잘라내는 위험이 있으므로 매 편집 후 잘림을 확인한다.
- **방식 B — 조각파일 + cat 조립(대 규모, 권장):** 셸(head+CSS+topbar / sidebar+`<main>` 열기 / footer+JS+`</main></div></body></html>`)과 콘텐츠를 **별도 조각 .html 파일**로 outputs 하위 build 폴더에 각각 Write한 뒤, bash `cat`으로 정해진 순서대로 이어붙여 최종 파일로 만든다. 각 조각은 128KB보다 훨씬 작아 Write 잘림이 원천 차단되고, 재빌드·부분 수정도 조각 단위로 안전하다. 예:
  ```bash
  cd <build 폴더>   # 예: outputs/pNNbuild
  cat 00a_head.html 00b_sidebar.html 10_home.html 20_*.html … 99_foot.html > "<결과 html 경로>"
  wc -c "<결과 html 경로>"; tail -c 40 "<결과 html 경로>"   # </html> 확인
  ```
  조각 파일명은 `00a_head`, `00b_sidebar`, `10_home`, `20_…`, `99_foot`처럼 정렬 가능한 접두어를 붙여 `cat` 순서를 자명하게 한다. 셸 3조각(head/sidebar/foot)은 `assets/template.html`을 플레이스홀더 치환해 나누되, footer·JS·`</html>`가 온전히 살아있는지 반드시 확인한다(잘린 참조 파일에서 셸을 뽑으면 안 됨).

`.screen-desc-list`는 6칸을 유지하되 화면 성격에 맞게 라벨을 조정한다. 프리셋:
- 입력/등록 화면: 목적 · 입력 필드 · 기능 · 검증 · 산출물 · 연계
- 조회 전용 화면: 목적 · 조회 조건 · 표시 항목 · 필터·정렬 · 산출물 · 연계
- 배치/처리 화면: 목적 · 입력·대상 · 처리 로직 · 검증·예외 · 처리 결과 · 연계

목업에 넣는 데이터는 **그 화면의 규칙을 증명하는** 값이어야 한다 — 차대균형이 핵심이면 합계행에 "차액 0", 자동반영이 핵심이면 해당 셀에 "→ 자동", 채번 규칙이 있으면 실제 형식의 번호. 빈 껍데기 목업은 피한다. md의 §번호를 crumb·패널 제목·표에 보존해 추적성을 유지한다.

**탭 화면은 모든 탭을 그린다.** `.tabbar`에 탭이 여러 개면 각 탭마다 `.tabpane`을 만들어 그 탭의 목업을 채운다(첫 탭만 그리고 나머지를 비워두지 않는다). 탭바 뒤에 `.mock`의 직속 자식으로 패널을 나열하고, 첫 패널만 `class="tabpane on"`, 나머지는 `class="tabpane"`. **탭 개수 = 패널 개수**이며 순서가 인덱스로 매칭된다. 스니펫·규칙은 `references/components.md` §4.4, 실제 2탭 화면의 전체 마크업은 `references/tab-example.md`. template.html의 CSS(`.tabpane`)와 탭 전환 JS가 인덱스 기반 패널 토글을 이미 처리하므로 마크업만 맞추면 된다.

**큰 파일 편집 주의.** 결과물이 수만 자(대략 128KB 초과)로 커지면 Edit/Write 도구가 파일 끝을 잘라내는 경우가 있다(뒷부분 섹션·푸터·스크립트 소실). 이 잘림을 겪었다면 방식 A를 고수하지 말고 방식 B(조각파일 + cat 조립)로 전환하는 게 가장 안전하다 — 각 조각이 작아 Write가 절대 잘리지 않는다. 부득이 단일 대용량 파일을 편집해야 하면 bash의 python(문자열 replace 후 파일 재기록)으로 처리하고, 편집 직후 `wc -c`·`</html>`·마지막 섹션 존재 여부를 확인한다.

### 6단계 — 검증한다 (필수)
결과 파일을 bash로 점검한다:
- 사이드바 `data-sec` 개수 = `<section id=` 개수, 그리고 값이 서로 1:1 매칭(누락/오탈자 없음).
- `class="section visible"`은 `#home` 하나뿐.
- `<section>`/`</section>`, `<div>`/`</div>` 태그 수 균형. `<main>`/`</main>` 각 1, `</html>` 1.
- 개발설계 md의 모든 화면 ID가 섹션으로 존재하는지 대조.
- `<span>`/`</span>` 개수도 대조한다. div 균형이 맞아도 `</span>`을 `</div>`로 잘못 쓴 인라인 태그 오타는 count 검사로 안 잡히니, 각 편집 직후 방금 쓴 블록을 Read로 눈으로 다시 확인한다.
- 탭 화면마다 **탭 개수 = `.tabpane` 개수**, 각 `.mock` 안 `.tabpane.on`은 정확히 1개인지 확인한다.
- 파일 크기(`wc -c`)와 마지막 줄(`</html>`)·마지막 섹션·푸터가 살아있는지 확인해 잘림이 없음을 보장한다.
- 참조 HTML에서 같은 EY 스타일 섹션을 원형 이식한 화면은 원본 섹션에서 `section id/class`만 치환한 문자열과 대상 섹션이 같은지 비교한다. 다르면 변경 지점을 설명할 수 있어야 하며, 특히 `titlebar`·`qbar`·`actionbar`·`table.grid`·`rpanel`·`statusbar` 구조가 달라지면 실패로 본다. 자체 스타일 참조 화면을 EY 스타일로 재작성한 경우에는 원본 업무 필드/검증/설명 항목이 누락되지 않았는지 대조한다.
가능하면 브라우저로 열어 섹션 토글과 탭 전환이 실제로 동작하는지 확인한다.

검증용 bash 예시:
```bash
f="<결과 html 경로>"
echo "ids:"; grep -oE 'id="[a-z0-9]+"' "$f" | sort -u | wc -l
echo "data-sec:"; grep -oE 'data-sec="[a-z0-9]+"' "$f" | sort -u | wc -l
echo "section open/close:"; grep -oE '<section' "$f" | wc -l; grep -oE '</section>' "$f" | wc -l
echo "div open/close:"; grep -oE '<div' "$f" | wc -l; grep -oE '</div>' "$f" | wc -l
echo "tabbar / tabpane / tabpane.on:"; grep -c 'class="tabbar"' "$f"; grep -oE 'class="tabpane' "$f" | wc -l; grep -oE 'class="tabpane on"' "$f" | wc -l
# data-sec 중 매칭되는 id가 없는 것 찾기
for d in $(grep -oE 'data-sec="[a-z0-9]+"' "$f" | sed -E 's/.*"([a-z0-9]+)"/\1/' | sort -u); do
  grep -q "id=\"$d\"" "$f" || echo "MISSING SECTION: $d"; done
echo "ending:"; tail -c 40 "$f"   # 방식 B(cat 조립) 후 </html> 온전성 확인
```

방식 B로 조립했다면 위 검사는 **최종 cat 결과물**에 대해 돌린다(조각 각각이 아니라). 태그 균형·탭=패널·data-sec↔id 매칭이 조각 경계를 넘어 전체에서 맞아야 한다.
