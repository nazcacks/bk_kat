BK 회계/세무 서비스 3계층 4단 메뉴구조도 V1.1

기준 문서: bk_설계서_v1.1.md · 이전본: 설계/화면설계/메뉴구조도_V1.0.HTML

목적: BK 플랫폼 → 복수 관리회사(OperatorOrg) → 관리대상회사(Tenant) 3계층을 전 메뉴에 적용하고, 플랫폼 자체의 사용자·내부조직·사용자그룹과 전체 업무 표준정보·메뉴 원본을 플랫폼콘솔에서 통합 관리하는 PF·OP·TN·CO 4채널 구조를 정의한다.

변경 기준: 설계/페이즈_화면/시스템관리자_화면_v1.0.HTML의 사용자·사용자그룹·그룹별 메뉴권한·메뉴관리 UX를 수용하되, 메뉴 원본 CUD·발행은 PF-04M, OP 발행본 조회는 OP-06D로 분리한다.

1. 표기 범례

PF-01 플랫폼콘솔 화면 BF-09 기초/기준정보 화면 JV-01 전표 입력 화면 AUX-02 보조부/보조원장 SA-VAT SA 기능 화면군 Step-up 재인증 필요 관리자 관리자/권한자 전용 기장 기장 모드 영향 선택 기능 플래그·업종·Phase별 노출 회색 들여쓰기: 화면 내 탭/하위 기능 또는 설계 추적 항목
메뉴는 제품 화면 기준으로 재구성했다. 설계서의 기술 장(데이터 모델, API, 배치, 검증, 보안, 운영)과 부록은 사용자 메뉴가 아니라 운영/관리/설계 추적 메뉴로 연결한다.

2. 3계층 운영모델

L1 · BK 플랫폼

platformId + platformOrgId 기준. PlatformOrg/Unit·PlatformUser·Membership·UserGroup·Role·DutyAssignment·SSO/SCIM, 복수 관리회사 생명주기, 전체 업무 표준정보와 PF/OP/TN/CO 메뉴 원본·버전·발행, 전역 정책·인프라를 관리한다. 플랫폼 Role은 테넌트 회계데이터 권한을 자동 상속하지 않는다.

L2 · 관리회사

operatorOrgId 기준. 자기 조직의 사용자·Membership·UserGroup·그룹별 OP 메뉴정책, 활성 수임관계·담당배정·세무대리 인증정보·기장 운영과 플랫폼 표준의 배포대상·일정·채택현황만 관리한다. 다른 관리회사 영역과 플랫폼 표준·메뉴 원본은 조회 권한을 넘어 변경할 수 없다.

L3 · 관리대상회사

tenantId 기준. 회계·세무 원천데이터의 소유자이며 자기 사용자·Membership·UserGroup·그룹별 TN 메뉴정책을 관리한다. 동일 시점 활성 PRIMARY 관리회사는 1개이며, 관리회사 이관 시 tenantId와 원천데이터는 유지한다.

관계·담당 분리

OperatorOrgTenantRelation은 회사 대 회사 수임관계이고, TenantAssignment와 BookkeepingAssignment는 직원 담당관계다. 두 관계를 분리해 계약·권한·책임시점을 추적한다.

운영자 테넌트 접근 게이트: 활성 OperatorOrgMembership ∧ 활성 PRIMARY 수임관계 ∧ 유효 담당배정 또는 접근세션 ∧ Role/Scope. 관리회사 종료는 테넌트 데이터 삭제를 의미하지 않는다.

3. 4채널 4단 메뉴 구조도

1뎁스 채널 → 2뎁스 상위 메뉴 → 3뎁스 하위 메뉴 → 4뎁스 세부 기능/화면
각 메뉴는 반드시 4단으로 해석한다. 1뎁스는 채널(플랫폼콘솔/관리회사 운영콘솔/테넌트 업무화면/공통), 2뎁스는 업무 영역, 3뎁스는 진입 메뉴, 4뎁스는 화면·탭·실행 기능이다.

7. 메뉴 운영 규칙

  • 채널 분리: PLATFORM 사용자는 PF, OPERATOR 사용자는 현재 operatorOrgId의 OP, TENANT 사용자는 자기 tenantId의 TN을 기본 채널로 사용한다. userGroup과 맞지 않는 채널은 미노출한다.
  • 플랫폼 콘솔 게이트: 활성 PlatformUser ∧ PlatformMembership ∧ 플랫폼 Role ∧ 허용 PlatformOrgUnit을 모두 검증한다. 사용자 계정, 조직소속, Role, DutyAssignment는 분리하며 변경 이력을 보존한다.
  • 3계층 컨텍스트: 모든 요청은 platformId를 기준으로 처리하고 OP는 operatorOrgId, TN은 tenantId를 필수로 한다. 관리회사 대행 TN 요청은 relationId와 assignmentId까지 검증한다.
  • 수임 접근 게이트: 활성 Membership ∧ 활성 PRIMARY 관계 ∧ 서비스 범위 ∧ 담당배정/승인 접근세션 ∧ Role/Scope를 모두 만족해야 한다.
  • 조직 격리: 관리회사는 다른 관리회사의 Membership·수임관계·테넌트 목록·사용량을 조회할 수 없다. 플랫폼 지원도 승인된 세션 범위 밖의 데이터는 접근할 수 없다.
  • 관리회사 이관: TRANSFER_IN_PROGRESS 동안 기존/신규 관리회사의 동시 쓰기를 금지한다. tenantId와 원천데이터는 유지하고 관계·배정·자격증명·미결업무를 컷오버한다.
  • 정책 상속: 보안·운영 정책은 GLOBAL → OPERATOR_ORG → PLAN → TENANT 순으로 병합하며 하위 계층은 상위의 강제/금지 규칙을 완화할 수 없다.
  • 표준 원본 단일화: 전체 업무 표준 원본·버전·검토·발행·폐기·롤백은 PF-04와 PLATFORM_STANDARD_ADMIN만 수행한다. OP-05는 PUBLISHED 표준을 읽고 배포대상·일정·사전검증·채택현황만 관리하며 원본 오버레이와 자체 버전을 만들 수 없다.
  • 메뉴 원본 단일화: PF/OP/TN/CO 메뉴 원본·다국어·route·MenuVersion CUD와 발행은 PF-04M에서만 수행한다. OP-06D는 PUBLISHED OP 메뉴 조회·영향 미리보기 전용이다.
  • 메뉴목록 기준: PF-04M은 트리를 탐색 보조로 사용하고 메뉴코드·상위메뉴·채널·레벨·유형·화면 ID·다국어명·route·flag·Phase·상태·source/detail의 목록 그리드를 기준으로 편집·검증·영향분석한다.
  • 그룹별 메뉴권한: PF-01P/OP-06P/TN-01P는 PUBLISHED 메뉴에 INHERIT/ALLOW/DENY를 적용한다. 하나라도 DENY이면 DENY, 부모 DENY는 하위에 전파하며 ALLOW만으로 Role VIEW/ENTER/ACTION을 생성하지 않는다.
  • 최종 권한평가: 활성 사용자 → Membership/scope → PUBLISHED MenuVersion → 그룹정책 → RoleMenuPermission → MenuActionGrant → data scope → 민감정보/Step-up → SOD 순으로 서버가 평가한다.
  • 미노출 원칙: 권한 없는 메뉴는 비활성이 아니라 미노출하고 API에서도 재차단한다. 요금제/기능 플래그 미포함 기능은 정책에 따라 업그레이드 안내로 노출할 수 있다.
  • 기장 모드: `TENANT_LED`는 이용회사 입력 중심, `OPERATOR_LED`는 운영자 기장 워크벤치와 회사 확인 화면 중심, `HYBRID`는 분담 규칙에 따라 입력/검토 권한을 나눈다.
  • 마감 상태: 메뉴는 유지하되 마감 기간 데이터의 신규/수정/삭제/기표 행위는 `CLOSING_LOCKED`로 차단한다.
  • 상태 불변성: `POSTED`, `CLOSED`, `LOCKED` 전표는 직접 수정/삭제하지 않고 역분개, 조정전표, 복구/이관 전용 메뉴로 처리한다.
  • 기능 플래그: 외화, 관계사, Cost-plus, 예산, 그룹보고, 제조원가/재고, XBRL, 다중 회계기준은 구독/업종/기능 플래그와 연동한다.
