## 1. 페이즈 개요

### 1.1 목표

Smart A Ch.20 데이터관리 7개 메뉴(SA-DAT-01~07)를 BK 멀티테넌시(SHARED+RLS)·회계기간 모델의 **운영 도구**로 재구성하고(§25.20, §25.0.2-C1), 그 하부 기반으로 플랫폼 백업·복구(PITR·테넌트 단위 복구·스냅샷, §2.4), 데이터 내보내기·보존·파기(§7.3), 데이터 티어링·콜드 스토리지 아카이빙(§7.5)을 함께 구축한다.

| 축 | 내용 | 근거 |
|---|---|---|
| 화면(메뉴) 축 | 삭제전표복구, 기수별 백업/복구, SmartA 내보내기(기수/월·분기), SmartA 올리기(전체기수/월·분기), 기수변경 — TN-15 데이터관리·이관 메뉴에 배치 | §25.20, §26.1-(2) TN-15 |
| 플랫폼 축 | 일 풀백업+WAL 기반 PITR, 테넌트 단위 논리 백업/복구, 마감·결산·신고 전 자동 스냅샷(`TenantSnapshot`), Hot/Warm/Cold 티어링, 보존연한·파기 워크플로 | §2.4, §7.3, §7.5, §14 |
| 무결성 축 | 회계 데이터 물리 삭제 금지 — 삭제전표복구는 `CANCELLED` 상태 전표의 복원 관점으로 설계(소프트삭제·결번 보존) | §16.1, §11.1(결번 관리), §12(전표 상태) |

### 1.2 범위 요약

- 1차: SA-DAT-01~07 **7개 메뉴 전부** + 백업/스냅샷/보존·파기/티어링 플랫폼 기반. 2차·3차 이월 항목 없음(§22.2).
- 이 페이즈는 타 페이즈 데이터(전표·기초코드·부가세 신고서식·조직)를 대상으로 동작하므로, 각 대상 페이즈의 데이터 모델 확정 시점에 맞춰 이관·복구 규칙을 점증 구현한다(횡단 병행의 사유).

### 1.3 MVP 연관

MVP 최우선은 페이즈 03 일반전표입력 + 페이즈 01 계정과목 관리이며(00_개요 §1), 페이즈 10은 **페이즈 01과 함께 다른 페이즈의 공통 기반으로 조기·횡단 착수** 대상이다(00_개요 §1 주석). 특히 SA-DAT-01(삭제전표복구)은 전표 입력(P03)의 소프트삭제 모델과 동시 설계가 필요하다(§25.20.1).

## 3. 업무 흐름

### 3.1 전표 삭제 → 복구 (SA-DAT-01)

```
[일반/매입매출전표입력(P03)]에서 전표 삭제
  → 물리 삭제 금지: status=CANCELLED 전이 + JournalHistory 보존 + 전표번호 결번 보존 (§16.1, §11.1)
  → [삭제전표복구 SA-DAT-01] 기간 조회(테넌트 RLS 범위) → 복구 대상 체크 → '전표복구' → 검토 → 확인
  → 새 전표번호 재채번(복합키 (tenant_id, 전표일자, 전표번호)) → 해당 입력 메뉴 재반영("복구전표" 표기) (§25.20.1)
  ※ 자동연동·복수거래·자금관리 전표는 복구 제외(조회 불가) (§25.20.1)
```

### 3.2 기수별 백업 → 복구 (SA-DAT-02)

```
회사 선택 → 모듈(회계) 선택 → 파일명 확인 → 암호 설정 → .wdb 암호화 백업 생성(다운로드)
복구: .wdb 선택 → 비밀번호 → 기수·사업자번호 무결성 키 검증 → 반영방법 선택
  ├ 신규 회사 생성 복구(신규 테넌트, "_복구회사" 명명)
  └ 현재 회사 복구(기존 테넌트 덮어쓰기: 해당 기수 기존 데이터 삭제 후 반영 + 기초코드 머지) (§25.20.2)
```

### 3.3 레거시 이관 (SA-DAT-03/04 내보내기 ↔ SA-DAT-05/06 올리기)

```
내보내기: BK → .whg(기수 단위, DAT-03) / .whgdata(월·분기+부가세 서식, DAT-04) → 레거시 Smart A [백업데이터복구]/[받은데이터올리기]
올리기: 레거시 Smart A [데이터백업 .sdb]/[보낼데이터만들기 .spwdata] → BK
  ├ DAT-05 전체기수: 기존 테넌트 데이터 전면 삭제·재구성 → 기수/인사년도 구성 → 부서/사원 자동 연결·수동 병합
  └ DAT-06 월·분기: 첨부 데이터 확인 → 기간/서식 단위 기존 데이터 삭제 후 적재 → 기초코드 머지 → 작업 이력(감사로그) (§25.20.5~6)
  ※ 공통 게이트: 사업자번호(·기수) 일치 검증, 암호 검증, 부서/사원 통합 미완료 시 진행 불가
```

### 3.4 기수변경 (SA-DAT-07)

```
회사정보 확인 → 데이터 존재 기수 확인(메뉴 실행 여부 기준) → 단일 기수 검증
  ├ 1개 기수: 변경 후 기수 입력 → 테넌트 기수 메타데이터(FiscalYear) 일괄 변경 → 기수변경이력(감사로그) 기록
  └ 2개 이상 기수: 변경 불가 → 수동 검토 워크플로(온라인 문의)로 분리 접수 (§25.20.7)
```

### 3.5 티어링·보존·파기 수명주기 (플랫폼)

```
[RDBMS Hot](최근 2개년) ─당해 마감─▶ [RDBMS Warm](3~5년 Read-Only) ─5년 초과 배치 이관─▶ [S3 Cold](Parquet+WORM)
콜드 이관 5단계: ①아카이브 파일 생성 → ②checksum/레코드수/잔액합계 검증 → ③ArchiveDataset 등록 → ④운영 파티션 read-only/분리 → ⑤조회 대상 제외 (§7.5)
파기: 보존연한 경과 → RetentionPolicy·개인정보 보유기간 확인 → 파기 승인 워크플로 → DataDestructionCertificate 발급 + DataDestructionLog 보존 (§7.3, §7.5)
```

## 4. 화면별 상세설계

### 4.1 삭제전표복구 (SA-DAT-01) — §25.20.1

- **목적**: [매입매출전표입력]·[일반전표입력]에서 삭제한 전표를 복구하여 해당 입력 메뉴로 재반영하는 전표 소프트삭제·휴지통 복구 도구. RLS 테넌트 범위 내 삭제 전표 보관(§25.20.1). BK에서 "삭제"는 물리 삭제가 아닌 `CANCELLED` 상태 보존이므로(§16.1, §12 전표 상태), 본 화면은 **CANCELLED 전표의 복원(재기표)** 화면이다.
- **진입경로**: TN-15 데이터관리·이관 > 삭제전표복구(§26.1-(2)).

**조회조건·입력 필드** (§25.20.1)

| # | 필드/조건 | 유형 | 필수 | 설명 |
|---|---|---|---|---|
| 1 | 전표 유형 탭 | 탭(매입매출/일반전표) | 필수 | 복구 대상 전표 종류 선택 |
| 2 | 회계기간(기수) | 선택 | 필수 | BK 조회조건 — 회계기간 단위 조회 |
| 3 | 사업장 | 선택 | 선택 | BK 조회조건 |
| 4 | 기간 | 일자 범위 | 필수 | 복구 대상 기간 조회 |
| 5 | 복구 대상 선택 | 행 체크박스 | 필수 | 복수 선택 가능 |
| 6 | 전표복구 버튼 | 실행 | — | 체크 후 '전표복구' → 검토 → '확인' 2단 확정 |

**기능·동작** (§25.20.1)

| # | 기능 | 동작 |
|---|---|---|
| 1 | 삭제 전표 조회 | 테넌트 RLS 범위 내 `CANCELLED` 전표를 유형·기간별 그리드 조회(가상 스크롤 — 공통 표준 UI) |
| 2 | 복구 실행 | 선택 전표를 원 입력 메뉴(일반/매입매출)로 재반영. 복구 시 **새 전표번호 재채번** — 복합키 `(tenant_id, 전표일자, 전표번호)` (원 번호는 결번 보존·재사용 금지, §11.1) |
| 3 | 복구 표기 | 복구된 전표는 입력 화면 가이드라인 영역에 "복구전표" 표기 |
| 4 | 복구 제외 필터 | 타 메뉴 전표전송분(신용카드·현금영수증·통장·회계전표처리·현금예금잔액검토·결산자료입력·정기청구입력 등), 매입매출 복수거래 설정 전표, 자금관리 설정 전표는 **조회 자체에서 제외**(복구 미지원) |
| 5 | 일괄삭제분 제외 | [현금및예금잔액검토(SA-CLS-10)] '전표추가'분의 Shift+F6 일괄삭제 전표는 복구 불가(§25.9 SA-CLS-10 처리규칙) |

**검증** (§25.20.1, §15, §16.1)

| # | 검증 | 처리 |
|---|---|---|
| 1 | 거래처코드가 [거래처등록]에 없는 전표 | 복구 차단(거래처 검증 선행) |
| 2 | 자동연동·자금관리 전표 | 무결성 보호를 위해 복구 제외 규칙 적용(조회 불가) |
| 3 | 복구 대상 일자 마감 여부 | 마감 잠금 인터셉터 — 마감 기간이면 403/비즈니스 예외(§16.1) |
| 4 | 전표번호 채번 충돌 | `NUMBERING_CONFLICT`(§13.3) |
| 5 | 복구 실행 멱등성 | `Idempotency-Key` — 동일 키 재요청 시 동일 응답(§13.3) |

- **산출물**: 복구된 전표(일반/매입매출). **연계**: [일반전표]·[매입매출전표](P03) 재반영, [거래처등록](P01) 검증. **권한**: `tenant:journal:write` scope + 메뉴 권한(공통 표준 §4.3). 복구 행위는 `DataChangeLog` 기록(공통 표준).

### 4.2 데이터백업 / 데이터복구(기수별) (SA-DAT-02) — §25.20.2

- **목적**: 기수(회계기간) 단위 회계관리 데이터의 테넌트별 백업·복구 도구. **기수·사업자번호를 백업 무결성 키**로 사용. 기수별데이터백업·기수별데이터복구 2탭 구성(§25.20.2). 플랫폼 관점에서는 §2.4 "테넌트 단위 복구(논리 백업 export/snapshot)"의 화면 구현.
- **진입경로**: TN-15 > 데이터백업/복구(기수별).

**입력 필드 — [탭1] 기수별데이터백업** (§25.20.2)

| # | 필드 | 유형 | 필수 | 설명 |

### 4.7 기수변경 (SA-DAT-07) — §25.20.7

- **목적**: 회계기수를 잘못 선택하여 데이터입력 작업을 한 경우 해당 기수로 바로잡는 회계기수(회계기간) 정정 기능. **한 개 기수만 있는 회사에만 지원**(2개 이상 기수 보유 회사는 추후/수동 검토). 변경 완료 후 데이터 확인 필요(§25.20.7). BK 모델에서는 Smart A "기수" 개념을 회계기간 표시·운영 속성으로 치환하되 **개업연도=1기 누적 기수 카운트·기수변경 운영 기능을 보존**한다(§25.0.2-C1).
- **진입경로**: TN-15 > 기수변경.

**입력 필드·화면 구성** (§25.20.7)

**기능·검증** (§25.20.7)

| # | 항목 | 처리 |

| 2 | 단일 기수 제약 | 1개 기수 외 다른 기수에서 메뉴 실행 시 "2개 이상 기수 데이터 존재"로 간주 → **기수변경 차단** |
| 3 | 다기수 처리 | 차단 시 [온라인 문의하기] 접수 — BK에서는 **수동 검토 워크플로**로 분리 |

## 11. 완료 기준(DoD) · 테스트 관점

### 11.1 완료 기준

1. SA-DAT-01~07 7개 화면이 §25.20의 입력 필드·기능·검증 전 항목을 충족하고 TN-15 메뉴로 노출된다(권한 미보유 시 미노출, §4.3).
2. 백업/PITR/스냅샷: SHARED 일 풀백업+PITR, 테넌트 단위 논리 백업, 마감/결산/신고 전 자동 스냅샷 동작(§2.4). SHARED RPO≤5분/RTO≤4시간 목표(§19.3 — 수치는 §23-21 확정 전 예시값).
3. 티어링: Hot/Warm/Cold 이관 5단계와 아카이브 검증 실패 시 중단·재처리 동작(§7.5, §15).
4. 보존·파기: read-only 보존, 파기 승인 워크플로·증명서·파기 로그(§7.3).
5. 모든 백업·복구·이관·기수변경 행위가 `DataChangeLog`(운영자는 `OperatorActivityLog`) append-only로 남는다(§17, §26.2).
