# BK 개발 상세설계서 - 관리회사 운영자

- 기준 문서: `bk_설계서_v1.1.md` (§1 시스템 개요, §3 인증, §4 권한·위임, §5 관리회사 접근 거버넌스, §6 표준 카탈로그, §7 구독·요금, §13 API, §14 배치, §18 알림, §21 기장 방식, §22 온보딩, §26 운영 콘솔)
- 작성일: 2026-07-08
- 대상 역할: 관리회사 소속 사원(`OperatorUser`). 운영 총괄, 운영 관리자, 고객 지원, 기준정보 운영, 기장 담당/검토/책임, 청구 담당, 감사자를 포함한다.
- 목적: 관리회사 운영자가 이용회사 온보딩·구독·지원·표준 배포·기장 업무·접근 거버넌스·고객 커뮤니케이션을 수행하기 위해 필요한 기능을 개발 설계 수준으로 정의한다.

## 1. 역할 정의

### 1.1 사용자 그룹

설계서 §1은 시스템을 두 사용자 그룹과 두 채널로 분리한다.

| 그룹 | 엔티티 | 채널 | 데이터 범위 |
|---|---|---|---|
| 관리회사 운영자 | `OperatorUser` | 운영 콘솔 OP-00~OP-11, 기장 컨텍스트 업무화면 | 배정 회사 또는 접근 세션 범위 |
| 이용회사 사용자 | `TenantUser` | 업무 화면 TN-00~TN-15 | 본인 소속 이용회사 |

관리회사 운영자는 전 이용회사에 대한 서비스 운영 권한을 갖지만, 모든 접근은 역할·담당 배정·접근모드·사유·로그로 통제한다. 회계 실무는 기장 계약이 있는 회사에서만 `BOOKKEEPING` 컨텍스트로 수행하며, 기장 계약이 없는 회사의 실무 개입은 `BREAK_GLASS` 예외 절차만 허용한다.

### 1.2 운영자 역할 체계

| 업무 역할 | 기술 Role 예시 | 핵심 책임 |
|---|---|---|
| 운영 총괄(슈퍼관리자) | `BK_MANAGER` 상위 정책 권한 | 전 운영 정책, 담당 배정, 접근 승인, 기장 책임, 표준·구독·테넌트 총괄 |
| 운영 관리자 | `BK_MANAGER` | 이용회사 등록, 계약·온보딩, 서비스 상태, 기장 모드 설정, 담당 배정 |
| 고객 지원 | 지원 전용 Role 또는 `BK_MANAGER` 제한 | 문의 대응, VIEW/SUPPORT 세션, 설정 지원, 고객 안내 |
| 기준정보 운영 | 표준 카탈로그 Role | 표준계정, 세율, 신고서식, 공통코드, 표준 Role·결재선 배포 |
| 기장 담당 | `BK_PREPARER` | 배정 회사 전표 작성, 증빙 매칭, 보조부 입력 |
| 기장 검토 | `BK_REVIEWER` | 배정 회사 전표 검토·승인, 마감 체크리스트 실행 |
| 기장 책임 | `BK_MANAGER` | 마감·결산·신고 승인, 담당 배정, 모드 전환 동의 |
| 청구 담당 | `BILLING` | 구독, 요금제, 청구, 수금, 미납 독촉 |
| 감사자 | `AUDITOR` | 접근·변경·기장·Break-glass 사후 검토 |

> `SEC_ADMIN` 중심의 인프라·배치·보안·KMS 업무는 `시스템관리자.md`에서 상세화한다. 본 문서는 관리회사 사원이 수행하는 업무 운영과 기장 운영을 중심으로 한다.

### 1.3 운영자 공통 원칙

- 권한 없는 메뉴는 비활성화가 아니라 미노출한다.
- 운영자는 운영 콘솔 로그인 시 2FA가 필수다.
- 테넌트 데이터 접근은 `VIEW`, `SUPPORT_SESSION`, `BOOKKEEPING`, `BREAK_GLASS` 접근모드 중 하나를 통해서만 가능하다.
- 담당 배정(`TenantAssignment`, `BookkeepingAssignment`)이 없는 회사는 검색·목록·작업에서 미노출하는 것을 기본으로 한다.
- 전표·결산·신고 등 회계 실무는 `OPERATOR_LED`/`HYBRID` 계약 및 배정이 있는 경우에만 허용한다.
- 모든 운영자 행위는 `AdminAccessLog`, `OperatorActivityLog`, `DataChangeLog` 등 해당 로그에 기록한다.

## 3. 접근모드와 세션

### 3.1 접근모드 정의

| 모드 | 사용 목적 | 운영자 가능 행위 | 통제 |
|---|---|---|---|
| `VIEW` | 고객 문의·현황 확인 | 마스킹 조회, 리포트 확인 | 사유, `AdminAccessLog`, 회사 통지 정책 |
| `SUPPORT_SESSION` | 설정·구성 지원 | 이용회사 설정 지원, 권한/환경 안내, 회계 실무 제외 | 시간제한, 사유, 로그, 회사 통지 |
| `BOOKKEEPING` | 기장 계약 회사의 상시 실무 | 전표 작성·검토·마감·결산·신고, 배정 Role 범위 | `BookkeepingConfig` + `BookkeepingAssignment`, 요약 통지 |
| `BREAK_GLASS` | 긴급 대행·예외 처리 | 기장 계약 없는 회사의 긴급 실무 포함 | 옵트인, 회사 통지/승인, 2인 승인, 녹화, 사후검토 |

### 3.2 세션 발급 조건

| 세션 | 발급 조건 | 만료 |
|---|---|---|
| `VIEW` | 운영자 권한 + 접근 사유 + 대상 회사 | 30~60분 |
| `SUPPORT_SESSION` | 운영자 권한 + 사유 + 지원 범위 | 30~60분, 연장 시 재사유 |
| `BOOKKEEPING` | 기장 모드가 `OPERATOR_LED`/`HYBRID` + 유효한 `BookkeepingAssignment` | 계약 기간 동안 유효, 배정 회사로 제한 |
| `BREAK_GLASS` | `BreakGlassConsent` + 요청 + 책임자 승인 + 범위·기간 | 승인된 기간 종료 시 자동 회수 |

### 3.3 화면 배너

운영자가 테넌트 컨텍스트에 진입하면 모든 화면 상단에 접근모드 배너를 고정 표시한다.

| 모드 | 배너 표시 |
|---|---|
| `VIEW` | 회사명, 조회 지원, 사유 |
| `SUPPORT_SESSION` | 회사명, 지원 세션, 잔여시간, 사유 |
| `BOOKKEEPING` | 회사명, 기장 모드, 담당 Role, 배정 범위 |
| `BREAK_GLASS` | 회사명, 긴급 대행, 녹화 중, 잔여시간, 승인번호 |

## 4. 운영자 공통 업무

### 4.1 운영 진입(OP-00)

**목적**: 운영자가 자신의 담당 회사와 미결 작업을 빠르게 확인하고 적정 접근모드로 진입한다.

| 기능 | 상세 |
|---|---|
| 담당 회사 검색 | `TenantAssignment`/`BookkeepingAssignment` 기반 목록. 미배정 회사 미노출 |
| 최근/즐겨찾기 | 최근 진입 회사, 즐겨찾기, 마감 임박 우선 표시 |
| 접근모드 선택 | VIEW/SUPPORT/BOOKKEEPING/BREAK_GLASS 중 가능 모드만 표시 |
| Step-up | 민감 회사, Break-glass, 신고 전송 승인 등 위험 작업 진입 전 재인증 |
| 작업 재개 | 미결 문의, 마감 체크리스트, 보고패키지, Ack 대기 건 표시 |

### 4.2 운영 현황(OP-01)

**목적**: 운영자가 전체 또는 담당 범위의 서비스 상태를 파악한다.

| 위젯 | 내용 |
|---|---|
| 온보딩 현황 | 가입대기, 표준 채택, 개시잔액 검증, ACTIVE 전환 대기 |
| 구독/요금 | 미납, 사용량 초과, 정지 예정 |
| 기장 현황 | 미처리 증빙, 미결 전표, 마감 D-day, 신고 기한 D-N |
| 고객 확인 | 마감/결산/신고/보고패키지 Ack 대기 |
| 배치/연계 | 수집 실패, DLQ, 외부 커넥터 만료 |
| 보안 이벤트 | 대량 다운로드, 권한 외 접근 시도, 비정상 로그인 요약 |

### 4.3 관리회사 조직·담당 배정(OP-02)

**목적**: 운영 조직과 이용회사 담당 관계를 관리한다.

| 기능 | 상세 |
|---|---|
| 운영 조직 조회 | `OperatorOrg`, `OperatorOrgUnit` 트리 표시 |
| 담당 테넌트 배정 | 운영자/조직 단위 `TenantAssignment` 등록, 유효기간, 역할 |
| 활성 담당 제한 | 동일 회사에 대한 활성 담당 관계는 원칙적으로 1건 |
| 기장 담당 연결 | `BookkeepingAssignment`는 OP-08과 연계해 작성/검토/책임 분리 |
| 활동 이력 | 배정 변경, 조직 변경, 접근 이력을 조회 |

**검증**:

- 조직 계층 순환참조 금지.
- 배정 시작일·종료일 역전 금지.
- 같은 회사 같은 역할의 활성 중복 배정 금지.
- 기장 작성자와 검토자는 동일인이 될 수 없다.

### 4.4 이용회사/테넌트 운영(OP-03)

**목적**: 이용회사 테넌트의 상태, 계약, 기장 모드 요약, 쿼터, 스냅샷, 운영 작업을 관리한다.

| 기능 | 상세 |
|---|---|
| 테넌트 목록 | 상태, 구독, 기장모드, 담당자, 최근 활동, 위험 플래그 |
| 테넌트 상세 | 사업자번호, 기본 설정, 격리 티어, 쿼터, 스냅샷, 기능 플래그 |
| 상태 변경 요청 | `PENDING`→`ACTIVE`, `SUSPENDED`, `DORMANT`, `TERMINATED` 전환 |
| 스냅샷 요청 | 마감/결산/신고 전 또는 운영 필요 시 `TenantSnapshot` 생성 요청 |
| 티어 전환 요청 | 대용량·보안 요구 회사의 `SCHEMA`/`DEDICATED` 전환 요청 |
| 쿼터 안내 | 사용자 수, 저장공간, API 호출, 월 전표 라인 수, 리포트 보관일 초과 안내 |

**주의**: 스냅샷 복구, 티어 전환, DB 조작은 시스템관리자 검증/승인 영역과 연결한다.

### 4.5 고객 커뮤니케이션

| 채널 | 기능 |
|---|---|
| 인앱 알림 | 접근 통지, 마감/신고 알림, Ack 요청, 미결항목 답변 |
| 이메일 | 보고패키지 전달, 장애 공지, 구독/청구 안내 |
| 포털 메시지 | Open Item 질의·증빙요청·이의 답변 |
| 웹훅 | 기업 고객의 외부 워크플로 연계 |

운영자는 법정 통지, 접근 통지, 보안 통지처럼 회사 정책상 차단할 수 없는 알림과 일반 안내 알림을 구분해야 한다.

## 5. 온보딩·구독·요금 업무

### 5.1 신규 이용회사 온보딩

설계서 §22의 신규 온보딩 절차를 운영자 업무 흐름으로 풀면 다음과 같다.

```mermaid
flowchart LR
  A["관리회사 준비"] --> B["표준 배포"]
  B --> C["테넌트 생성"]
  C --> D["구독 개설"]
  D --> E["기장 모드 설정"]
  E --> F["사용자·인증 정책"]
  F --> G["역할·결재선·배정"]
  G --> H["표준 채택·차원·알림"]
  H --> I["기준정보·개시잔액 적재"]
  I --> J["검증 후 ACTIVE"]
```

| 단계 | 운영자 기능 | 차단 조건 |
|---|---|---|
| 관리회사 준비 | 담당 조직·담당자 지정 | 담당 공백 |
| 표준 배포 | 표준계정·세율·역할·서식 기본 세트 적용 | 필수 표준 미적용 |
| 테넌트 생성 | 사업자번호, 회사명, 격리 티어, 기본 설정 | 활성 테넌트 사업자번호 중복 |
| 구독 개설 | 요금제, 결제 조건, 사용량 쿼터 | 요금제 미선택 |
| 기장 모드 설정 | `TENANT_LED`/`OPERATOR_LED`/`HYBRID`, 주기장, 계약 참조 | 상대방 동의 없음 |
| 사용자·인증 | 회사 관리자 초대, 2FA 하한 적용 | 관리자 없음 |
| 역할·결재선 | 표준 Role 채택, 결재선 구성 | 결재선 미설정 |
| 표준·차원·알림 | `DimensionConfig`, 알림정책, 표준 채택 | 필수 차원 불일치 |
| 개시잔액 | 전기잔액·거래처·고정자산·어음 이관 | 차대변 불균형 |
| ACTIVE 전환 | 업무 시작 | 검증 게이트 미통과 |

### 5.2 구독 수명주기(OP-04)

| 상태 | 운영자 처리 |
|---|---|
| `TRIAL` | 체험 한도 안내, 정식 전환 유도 |
| `ACTIVE` | 정상 운영, 사용량 모니터링 |
| `PAST_DUE` | 미납 독촉, 납부 안내, 정지 예정 공지 |
| `SUSPENDED` | 조회만 허용, 신규 처리 차단, 복구 조건 안내 |
| `GRACE` | 데이터 내보내기 지원, 재계약 안내 |
| `TERMINATED` | 보존기간 내 조회/다운로드, 파기 일정 안내 |
| `REACTIVATED` | 보존 데이터 복원·재활성화 확인 |

### 5.3 청구·수금

| 기능 | 상세 |
|---|---|
| 요금제 관리 | 기본료, 사용자 수, 전표 건수, 저장용량, 전자세금계산서 건수 |
| 사용량 조회 | `UsageMeter` 기준 월별 사용량 |
| 청구서 발행 | `Invoice`, 세금계산서/영수증 연계 |
| 수금 관리 | `Payment`, 미수, 환불, 결제 실패 |
| 독촉 | `DunningCase` 단계별 안내, `PAST_DUE`/정지 예약 |
| 다운그레이드 영향 | 쿼터·기능 플래그 축소 영향 사전 고지 |

**통제**:

- 카드번호 등 결제수단 원문은 저장하거나 대행 입력하지 않는다. PG 토큰 참조만 저장한다.
- 청구 이력은 append-only로 보존한다.

## 6. 공통 표준 카탈로그 운영(OP-05)

### 6.1 표준 유형

| 표준 | 운영자 작업 |
|---|---|
| `STD_ACCOUNT` | 업종별 표준계정 템플릿 버전 관리 |
| `STD_VAT` | 부가세 세율·과세유형 변경 배포 |
| `STD_TAX_RULE` | 세법 기준값, 적용일자, 종료일 관리 |
| `STD_WITHHOLDING` | 원천세 기준, 간이세액표, 지방소득세율 |
| `STD_FORM` | 부가세/법인세/원천세 신고서식 버전 |
| `STD_CODE` | 결제수단, 거래유형 등 공통코드 |
| `STD_ROLE` | 표준 권한 묶음 |
| `STD_APPROVAL` | 표준 결재선 템플릿 |
| `STD_REPORT_PACKAGE` | 월별 보고패키지 템플릿 |
| `STD_GROUP_MAPPING` | 그룹보고 매핑 |
| `STD_ANOMALY_RULE` | 이상탐지 규칙·임계치 |

### 6.2 배포 모드

| 모드 | 운영 규칙 |
|---|---|
| `MANDATORY` | 강제 적용. 회사 거부 불가. 세율·신고서식 등 |
| `RECOMMENDED` | 권고. 회사 채택/보류 선택, 미채택 알림 |
| `OPTIONAL` | 선택. 회사 필요 시 채택 |

### 6.3 표준 배포 흐름

1. 새 표준 버전을 `DRAFT`로 작성한다.
2. 변경 diff와 영향 회사를 산정한다.
3. 회귀 검증을 수행한다.
4. 운영 총괄 승인 후 `PUBLISHED`한다.
5. 배포 모드에 따라 회사별 적용 또는 채택 알림을 발송한다.
6. 오류 발생 시 `StandardRollbackLog`와 함께 직전 버전으로 롤백한다.

**검증**:

- 회사 확장 항목과 표준 항목 충돌 시 `STANDARD_CONFLICT`.
- 강제 표준은 적용 시작일과 회계기간 정합성을 검증한다.
- 표준 배포 후 권한 평가 캐시와 메뉴 캐시를 무효화한다.

## 7. 사용자·권한·가드레일 지원(OP-06)

### 7.1 이용회사 위임관리 지원

이용회사 회사 관리자는 자사 사용자·커스텀 Role·결재선·환경설정을 위임관리한다. 운영자는 정책 하한과 가드레일을 관리·지원한다.

| 기능 | 운영자 역할 |
|---|---|
| 사용자 초대/잠금해제 지원 | 회사 관리자 요청 기반 지원, 사유·로그 기록 |
| 2FA 초기화 지원 | 본인확인 후 초기화, 회사 정책 하한 준수 |
| 커스텀 Role 검토 | 표준 Role 복제·확장 안내, 위험 권한 경고 |
| 결재선 구성 지원 | 표준 결재선 템플릿 적용, SOD 검증 |
| 메뉴·행위권한 | `RoleMenuPermission`, `MenuActionGrant` 기준으로 노출/행위 권한 점검 |

### 7.2 정책 가드레일

| 가드레일 | 운영자 기능 |
|---|---|
| 인증 하한 | 2FA 필수, 비밀번호 강도 약화 차단 |
| 권한 상한 | 위험 권한 부여 제한, 단일 사용자 최대 Role 수 |
| 사용자 수 | 요금제별 사용자/관리자 수 제한 |
| SOD | 작성-승인 분리 비활성화 금지 |
| 커스텀 Role 한도 | 회사별 최대 개수 제한 |

가드레일 유형은 `FORBID`, `REQUIRE_SOD`, `REQUIRE_STEPUP`, `REQUIRE_APPROVAL`을 사용한다.

## 8. 고객 지원 업무

### 8.1 VIEW 세션

**사용 예시**: 고객 문의로 화면 설정, 권한 상태, 마감 상태, 수집 실패 원인을 확인할 때.

| 규칙 | 내용 |
|---|---|
| 데이터 표시 | 기본 마스킹 |
| 입력 | 회계 데이터 CUD 불가 |
| 필수 | 접근 사유, `AdminAccessLog` |
| 통지 | 회사 정책에 따라 실시간 또는 요약 통지 |

### 8.2 SUPPORT_SESSION

**사용 예시**: 고객과 함께 환경설정, 알림 설정, 권한 템플릿, 커넥터 연결 상태를 조정할 때.

| 가능 | 불가 |
|---|---|
| 설정 지원, 구성 변경, 권한/메뉴 안내, 연계 상태 확인 | 전표 작성, 승인, 기표, 마감, 신고 전송, 개인정보 평문 조회 |

SUPPORT_SESSION은 시간제한이 있으며 연장 시 사유를 다시 입력한다.

### 8.3 Break-glass 요청

기장 계약이 없는 `TENANT_LED` 회사에서 긴급 실무 처리가 필요한 경우에만 사용한다.

```mermaid
flowchart LR
  A["운영자 요청"] --> B["사유·범위·기간 입력"]
  B --> C["관리회사 책임자 승인"]
  C --> D["회사 통지/승인 옵션"]
  D --> E["세션 시작·녹화"]
  E --> F["작업 수행"]
  F --> G["자동 회수"]
  G --> H["사후검토 리포트"]
```

**필수 조건**:

- 회사의 `BreakGlassConsent`가 활성화되어 있어야 한다.
- 사유, 대상 범위, 기간, 예상 조치가 명확해야 한다.
- 2인 통제를 적용한다.
- 모든 조작은 녹화 및 `AdminAccessLog`/`DataChangeLog`에 기록한다.

## 9. 기장 운영(OP-08)

### 9.1 기장 모드

| 모드 | 전표 입력 | 마감·결산·신고 | 이용회사 역할 | 관리회사 역할 |
|---|---|---|---|---|
| `OPERATOR_LED` | 관리회사 기장 담당 | 관리회사 | 증빙 제출, 거래 설명, 결과 확인, 신고 승인 | 기장 대행 전체 |
| `TENANT_LED` | 이용회사 | 이용회사 | 전체 자체 수행 | 운영·지원만 |
| `HYBRID` | 양측 분담 | 주기장 측 | 분담 입력 + 확인 | 분담 입력 + 주기장 업무 |

### 9.2 기장 담당 배정

| Role | 권한 | SOD |
|---|---|---|
| `BK_PREPARER` | 배정 회사 전표 작성, 증빙 매칭, 보조부 입력 | 검토/마감 승인자와 분리 |
| `BK_REVIEWER` | 전표 검토·승인, 마감 체크리스트 실행 | 작성자와 동일인 금지 |
| `BK_MANAGER` | 마감·결산·신고 승인, 담당 배정 관리 | 신고 전송 Step-up |

`BookkeepingAssignment(tenantId, operatorUserId, role, validFrom/To)`가 없으면 `BOOKKEEPING` 컨텍스트 진입은 `ASSIGNMENT_REQUIRED`로 차단한다.

### 9.3 기장 워크벤치

| 기능 | 상세 |
|---|---|
| 담당 회사 보드 | 미처리 증빙, 미결 전표, 마감 D-day, 신고 기한, Ack 대기 |
| 컨텍스트 진입 | 회사 카드 선택 → `BOOKKEEPING` 배너와 함께 업무 화면 진입 |
| 일괄 작업 | 미처리 증빙 큐, 마감 체크리스트 일괄 실행, 확인 요청 발송 |
| 보고패키지 | 월별 결산 보고패키지 조립·검증·전달·Ack |
| Open Items | 질의·증빙요청·이의사항 등록/답변/해결 |
| 인계 | 담당 변경 시 미처리 목록·노트 자동 생성 |

### 9.4 OPERATOR_LED 업무 흐름

```mermaid
flowchart LR
  A["이용회사 증빙 제출"] --> B["BK_PREPARER 전표 작성"]
  B --> C["BK_REVIEWER 검토·승인"]
  C --> D["기표"]
  D --> E["월차 마감"]
  E --> F["보고패키지 전달"]
  F --> G["이용회사 Ack"]
  G --> H["결산·신고"]
  H --> I["신고 전 이용회사 승인"]
  I --> J["전자신고 전송"]
```

### 9.5 HYBRID 업무 흐름

HYBRID는 `BookkeepingScopeRule`로 입력 주체를 나눈다.

| 분담 기준 | 예시 |
|---|---|
| `JOURNAL_TYPE` | 매입은 이용회사, 결산분개는 관리회사 |
| `TRANSACTION_SOURCE` | 카드/은행 자동수집분은 관리회사 검토 |
| `ACCOUNT_RANGE` | 급여·세무 민감 계정은 관리회사 |
| `MODULE` | 고정자산·법인세·부가세 신고는 관리회사 |

**충돌 방지**:

- owner 위반 저장 차단.
- 동일 거래 양측 입력 시 중복 경고와 이상탐지 reason code `DUP_CROSS_PARTY`.
- 미정의 영역은 주기장 측에 귀속한다.

### 9.6 이용회사 확인(Ack)

| 대상 | 상태 | 처리 |
|---|---|---|
| 월차 마감 | `PENDING` → `CONFIRMED`/`OBJECTED` | 이의 시 보완 후 재요청 |
| 보고패키지 | `DELIVERED` → `ACKED` | 질의는 Open Item 등록 |
| 신고 전송 | 승인 전 Step-up | 승인 이력 신고 로그와 연결 |

`ackPolicy=REQUIRED`이면 확인 완료 전 다음 단계(예: 신고 생성/전송)를 차단한다.

## 10. 신고·마감·보고 통제

### 10.1 주기장 전속 권한

다음 행위는 주기장(`primaryBookkeeper`) 측만 수행한다.

| 행위 | 비주기장 시도 |
|---|---|
| 가마감/본마감/마감해제 | `NOT_PRIMARY_BOOKKEEPER` |
| 결산정리분개·재무제표 확정 | 차단 |
| 현금흐름 조정 승인 | 차단 |
| 세무 신고서 생성·전자신고 전송 | 차단 |
| 공시 생성·제출 | 차단 |
| 개시잔액·변환 반영 승인 | 차단 |

### 10.2 신고 전송 승인

주기장=OPERATOR인 신고는 관리회사가 세무대리인 자격으로 수임한 경우에만 가능하다. `filingApprovalRequired=Y`이면 다음 절차를 따른다.

1. 신고서 요약(과세표준, 세액, 제출 대상)을 이용회사 승인자에게 전달한다.
2. 이용회사 승인자가 Step-up 인증 후 승인한다.
3. 관리회사 기장 책임자가 신고 전송 Step-up을 수행한다.
4. 전송 파일 해시, 승인자, 전송자, 시각을 감사로그에 연결한다.

### 10.3 보고패키지

| 단계 | 처리 |
|---|---|
| 생성 | 템플릿 기준으로 재무제표, 시산표, 원장, 계정별 명세, 세무 집계 등 조립 |
| 검증 | 계정별 tie-out, 미결항목 0건, 마감 상태 확인 |
| 전달 | 포털/이메일로 이용회사 전달 |
| 확인 | Ack 또는 Objected |
| 보완 | 이의·질의는 Open Item으로 추적 |

- 운영자 API는 `userGroup=OPERATOR`, Role, scope, 담당 배정, 접근모드를 검증한다.

## 15. 개발 수용기준

### 15.1 운영 콘솔

- 운영자는 담당 배정 또는 Role 권한에 맞는 회사와 메뉴만 볼 수 있다.
- 접근 세션 없이 테넌트 데이터 화면에 진입할 수 없다.
- 접근모드 배너가 모든 테넌트 컨텍스트 화면에 고정 표시된다.
- VIEW/SUPPORT 세션에서는 회계 실무 CUD API가 차단된다.

### 15.2 온보딩·구독

- 신규 회사는 필수 온보딩 게이트를 통과해야 `ACTIVE`로 전환된다.
- 개시잔액 차대변 불균형이면 전환이 차단된다.
- 미납 상태 전환은 구독 상태와 테넌트 상태를 분리해 기록한다.
- 결제수단 원문은 저장되지 않는다.

### 15.3 표준 카탈로그

- 표준 배포 전 diff와 영향 회사가 표시된다.
- `MANDATORY` 표준은 적용일 기준 자동 반영된다.
- 표준 충돌은 저장 또는 배포를 차단한다.
- 롤백 이력이 보존된다.

### 15.4 기장 워크벤치

- `BookkeepingAssignment` 없는 운영자는 담당 회사 보드를 볼 수 없다.
- `BK_PREPARER`와 `BK_REVIEWER`가 동일인이면 승인 흐름이 차단된다.
- `OPERATOR_LED` 회사의 이용회사 전표 입력 메뉴는 정책에 따라 비활성 또는 요청 전용으로 전환된다.
- 보고패키지는 검증 완료 후 전달 가능하고 Ack 이력이 기록된다.
- 신고 전송은 이용회사 승인과 Step-up 없이 실행되지 않는다.

### 15.5 감사·통제

- 운영자 접근은 `AdminAccessLog`에 사유와 함께 기록된다.
- 개인정보 평문 조회는 Step-up과 `PersonalDataAccessLog`가 없으면 차단된다.
- Break-glass는 사전동의, 2인 승인, 녹화, 사후검토 없이 완료될 수 없다.
