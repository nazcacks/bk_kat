## 1. 범위와 공통 원칙

운영자콘솔은 현재 `operatorOrgId` 안에서 수임한 테넌트를 운영하는 채널이다. 플랫폼 전역 원본(PF)과 이용회사 회계업무(TN)의 경계를 넘지 않는다.

1. GNB와 모든 쿼리의 최상위 scope는 인증된 operatorOrgId로 고정한다.
2. 테넌트 접근은 활성 Membership, 활성 수임관계, serviceScopes, 담당배정 또는 승인 접근세션, Role/data scope를 모두 만족해야 한다.
3. 다른 관리회사의 조직·사용자·수임관계·테넌트·사용량은 미노출하고 API에서도 차단한다.
4. 회계 실무는 `BOOKKEEPING` 모드와 `BookkeepingAssignment`가 있는 테넌트에서만 수행한다.
5. `VIEW/SUPPORT/BOOKKEEPING/BREAK_GLASS` 접근모드마다 목적·허용행위·시간·통지를 분리한다.
6. 플랫폼 표준과 메뉴 원본은 OP에서 변경하지 않는다. OP-05는 배포, OP-06-06-01은 메뉴 조회만 수행한다.
7. 권한 없는 메뉴는 미노출하며 모든 중요행위는 operatorOrgId/tenantId/relationId/assignmentId를 감사키로 남긴다.

## 3. 공통 화면·API 표준

- GNB: 현재 관리회사명/operatorOrgId/조직 Role을 고정하고 수임회사 선택기를 제공한다.
- 테넌트 진입 시 관리회사명·테넌트명·relationId·접근모드·담당 Role·만료시각을 배너로 표시한다.
- 목록은 operatorOrgId 서버 필터를 제거할 수 없고 담당범위 필터를 추가 적용한다.
- 저장은 `If-Match`, `Idempotency-Key`, reason을 사용하고 고위험 action은 Step-up·승인을 요구한다.
- 개인정보는 기본 마스킹하며 평문 보기는 재인증·사유·시간제한·`PersonalDataAccessLog`를 요구한다.
- API prefix: `/api/op/v1`; tenantId만으로 접근하지 않고 relationId/assignmentId 또는 accessSessionId를 함께 검증한다.

## 5. 메뉴별 상세 기능

### 5.1 OP-00 운영 진입

수임회사 선택은 현재 operatorOrgId의 활성 PRIMARY 또는 허용 SECONDARY 관계와 담당배정을 모두 만족하는 테넌트만 표시한다. 검색, 최근 접근, 즐겨찾기, 관계/서비스범위/담당자 요약을 제공한다.

접근모드별 통제:

| 모드 | 용도 | 필수조건 | 허용 범위 |
|---|---|---|---|
| VIEW | 마스킹 조회 | 활성 관계·Role·사유 | 조회만 |
| SUPPORT | 설정/장애지원 | serviceScopes·시간제한·통지 | 승인된 설정 action |
| BOOKKEEPING | 기장 실무 | 기장계약·모드·BookkeepingAssignment | TN 업무 작성/검토 범위 |
| BREAK_GLASS | 긴급대행 | 사전동의·2인승인·Step-up·녹화 | 승인 화면/action/시간 |

세션에는 operatorOrgId, tenantId, relationId, assignmentId, mode, reason, expiresAt을 저장한다. 관계·배정 종료나 Role 변경 시 즉시 무효화한다.

API: `/tenants/selectable`, `/favorites`, `/access-sessions`, `/access-sessions/{id}/terminate`.

### 5.2 OP-01 운영 현황

- 대시보드: 수임회사·온보딩·구독·수집/배치·보안사건·미결기장·마감/신고 임박·Open Items를 Role/담당범위로 집계한다.
- 운영 리포트: 고객별 사용량, 오류율, 처리시간, SLA/SLO, 반복장애, 담당 부하를 기간 비교한다.
- 다른 관리회사의 집계는 포함하지 않으며 drill-down도 동일 operatorOrgId를 강제한다.
- 위젯의 기준시각, 집계지연, 데이터 출처를 표시한다.

API: `/dashboard`, `/reports/operations`, `/reports/sla`, `/alerts`.

### 5.3 OP-02 우리 조직·소속

- 조직 기본정보: 현재 OperatorOrg 프로필과 조직트리만 CUD한다. 다른 관리회사 생성·상태변경은 PF-01 전용이다.
- Membership: 초대, 조직단위, Role, 유효기간, 휴면·종료, SSO/SCIM 상태를 관리한다. 종료 시 세션·배정을 재검증한다.
- 담당배정: 활성 관계에 일반 담당과 기장 작성/검토 담당을 배정한다. Membership와 relationId의 조직 일치, 유효기간, 작성자/검토자 SOD를 검증한다.
- IP 허용목록: CIDR 중복/겹침과 현재 접속 IP 제외를 경고하며 변경사유·Step-up을 요구한다.

API: `/organization`, `/org-units`, `/memberships`, `/tenant-assignments`, `/bookkeeping-assignments`, `/console-ip-allowlist`.

### 5.4 OP-03 수임회사·테넌트

- 목록은 활성 관계, serviceScopes, 온보딩, 기장모드, 담당자, 구독, 위험 플래그를 제공한다.
- 상세 운영은 허용 scope 안에서 격리티어, 쿼터, 스냅샷, 백업상태, 비동기 리포트 job을 조회·요청한다.
- 티어 전환·복구 같은 고위험 인프라 작업은 Step-up, 사전검증, 승인, 작업창을 요구한다.
- 수임종료·서비스범위 변경·관리회사 이관은 요청만 생성하고 PRIMARY 변경/이관 승인은 PF-03에서 수행한다.

API: `/tenants`, `/tenants/{id}`, `/tenants/{id}/snapshots`, `/tenant-jobs`, `/relation-change-requests`, `/transfer-requests`.

### 5.5 OP-04 구독·요금·사용량

- `TenantServiceContract`에 기장·세무·급여·보고 serviceScopes, 요금, 효력기간, 기장모드를 연결한다.
- `UsageMeter`를 계약/테넌트별로 집계하고 Invoice 발행, Payment 반영, 환불, Dunning 단계를 관리한다.
- `TenantDirectSubscription`은 조회전용으로 표시해 플랫폼 직접구독과 관리회사 서비스계약을 혼합 청구하지 않는다.
- 미납 상태는 안내→납부유예→제한예고→정지요청 순이며 원천데이터 삭제는 금지한다.
- 청구 이력은 append-only, 결제수단은 PG token 참조만 저장한다.

API: `/service-contracts`, `/usage`, `/invoices`, `/payments`, `/dunning-cases`, `/direct-subscriptions`.

### 5.6 OP-05 플랫폼 표준 배포

- PUBLISHED 플랫폼 표준의 유형, 버전, 효력일, 법적근거, 변경요약, diff를 읽기전용 조회한다.
- 현재 operatorOrgId와 활성 수임관계 테넌트만 배포대상으로 선택한다.
- 배포정책은 대상 조건, 권고/강제 일정, 사전검증, 알림, 재시도 정책을 관리한다.
- 채택현황은 테넌트 채택 versionId, 회사계정/항목 매핑, 지연, 충돌, 실패사유를 보여주고 재안내한다.
- 원본 복제, 오버레이, 자체 버전, 플랫폼 발행본 강제롤백은 금지한다.

API: `/standards/published`, `/standard-distribution-policies`, `/standard-distributions`, `/standard-adoptions`.
### 5.7 OP-06 사용자·인증·권한

#### 사용자·그룹

- OP-06-01-01은 현재 operatorOrgId 사용자만 조회·초대·정지·종료하고 소속/Role, MFA, 신뢰기기, 로그인/활성세션을 관리한다.
- 플랫폼 사용자와 타 관리회사 계정은 변경할 수 없다. loginId 전역 중복, Membership scope, 자기승격을 검증한다.
- OP-06-02-01은 그룹 기본정보와 구성원을 별도 저장한다. 그룹코드 유일, 상위그룹 순환, 타 조직 구성원, 기간중복을 차단한다.
- OP-06-03-01은 PUBLISHED OP 메뉴에 INHERIT/ALLOW/DENY를 일괄 저장하고 사용자·시점별 최종권한과 근거를 미리보기한다.

#### 인증·Role·메뉴조회

- OP-06-04-01은 MFA, SSO, 패스키, 세션, 신뢰기기, Step-up, IP 정책을 관리하되 GLOBAL 하한을 완화하지 못한다.
- OP-06-05-01은 조직 Role, 기장 Role, data scope, 결재선, PolicyGuardrail을 관리한다. 자기승격·SOD·계약/관계/배정 기간을 검증한다.
- OP-06-06-01은 PUBLISHED OP 메뉴, route/feature flag, Role/그룹 영향을 조회·출력한다. 메뉴 원본 CUD와 발행은 PF-04-03-01 전용이다.

API: `/users`, `/user-security-actions`, `/user-groups`, `/group-members`, `/group-menu-permissions`, `/permission-simulations`, `/auth-policies`, `/roles`, `/approval-lines`, `/menus/published`.

### 5.8 OP-07 접근 거버넌스

- 접근세션 목록은 대상, 모드, 사유, 승인, 시작/만료, 수행 action, 종료사유를 제공한다.
- 개인정보 평문 열람과 대량 출력은 세션 안에서도 별도 Step-up과 `PersonalDataAccessLog`가 필요하다.
- Break-glass는 활성 수임관계와 사전동의를 요구하며 요청자/승인자/사후검토자를 분리한다.
- 승인 범위를 화면·API·action·시간·레코드 범위로 고정하고 전 세션을 녹화한다.
- 플랫폼 지원 요청은 관리회사에서 승인/거절하고 시작·종료 통지와 행위로그를 검토한다.

API: `/access-sessions`, `/break-glass/requests`, `/break-glass/recordings`, `/platform-support-sessions`, `/access-reviews`.

### 5.9 OP-08 기장 워크벤치

기장모드는 관계별로 관리한다.

| 모드 | 주 작성자 | 관리회사 기능 | 이용회사 의무 |
|---|---|---|---|
| TENANT_LED | 이용회사 | 검토·지원 중심 | 입력·마감·신고 책임 |
| OPERATOR_LED | 관리회사 | 작성·검토·마감·보고 | 중요 산출물 확인/Ack |
| HYBRID | 분담규칙 | 배정업무 작성·검토 | 자사 분담업무와 확인 |

모드변경은 양측 동의, 효력일, 미결업무 영향, 담당 재배정, Step-up을 요구하며 소급 적용하지 않는다.

담당회사 보드는 활성 관계·serviceScopes·기장모드·BookkeepingAssignment를 모두 만족하는 회사만 표시하고 전표, 자동수집, 장부, 마감, 신고, 보고 TN 화면으로 BOOKKEEPING 컨텍스트를 발급한다.

보고·미결 큐는 ReportPackage 조립, 계정명세/tie-out, Open Item, 증빙요청, 고객 Ack, 법정기한을 관리한다. 기장 인계는 담당변경 또는 PF-03 이관 시 미결전표·질의·마감·신고·보고·시크릿 회수 결과와 책임시점을 기록한다.

API: `/bookkeeping-configs`, `/bookkeeping-mode-change-requests`, `/bookkeeping-workbench`, `/report-packages`, `/open-items`, `/bookkeeping-handovers`.

### 5.10 OP-09 운영·배치·연계

- 범위형 커넥터는 scopeType(OPERATOR_ORG/TENANT), scopeId, 유형, 상태, 마지막 성공/실패, 인증서 만료를 관리한다.
- 시크릿 원문은 표시하지 않고 KMS secretRef만 사용한다.
- 자격증명 위임은 목적, relationId, serviceScopes, 기간, 허용 커넥터를 명시하고 관계 종료·이관 시 즉시 회수한다.
- 배치 화면은 ETL/수집/보고 job, 큐, worker, 재시도, DLQ, SLO, 장애를 현재 조직과 수임범위로 조회한다.
- 재처리는 동일 멱등키·요청해시와 실행 직전 관계/배정/상태 재검증을 요구한다.

API: `/connectors`, `/credential-delegations`, `/job-runs`, `/dlq`, `/job-runs/{id}/retry`, `/slo`, `/incidents`.

### 5.11 OP-10 보안·감사 운영

- 조직 보안은 KMS key version/회전, DLP 이벤트, 대량행위, 취약점, 인증 이상, 보존/파기 점검을 운영한다.
- 키 회전 권한과 평문 접근 권한을 분리하고 회전·폐기는 Step-up과 승인 증적을 요구한다.
- 관계·이관 감사는 수임관계, 담당배정, 접근세션, 기장 인계, 관리회사 이관의 전후값과 책임시점을 읽기전용으로 제공한다.
- 감사로그와 해시체인은 append-only이며 export도 사유·승인·만료 URL로 통제한다.

API: `/security/events`, `/kms/keys`, `/dlp-events`, `/vulnerabilities`, `/audit/relations`, `/audit-chain/verify`, `/audit-exports`.

### 5.12 OP-11 AI·규칙 운영

- 플랫폼 발행 이상탐지 규칙을 수임 테넌트에 배포하고 허용 임계치, 모델 버전, 검토 큐, 피드백 상태를 운영한다.
- 규칙 원본은 수정하지 않고 테넌트별 기준선·학습데이터·피드백을 격리한다.
- 자연어 조회는 허용 QueryTool, 읽기전용 범위, 행/금액/기간 제한, 민감필드 마스킹, prompt/tool/output 로그를 관리한다.
- AI 결과는 자동 기표·신고·마감하지 않고 담당자의 명시적 검토결정을 요구한다.

API: `/ai/rules/published`, `/ai/rule-distributions`, `/ai/models`, `/ai/review-queue`, `/ai/guardrails`, `/ai/query-tool-logs`.

## 6. 핵심 워크플로

### 6.1 수임회사 접근

```mermaid
flowchart LR
  A[활성 Membership] --> B[수임관계 조회]
  B --> C[serviceScopes 확인]
  C --> D[담당배정/승인세션 확인]
  D --> E[접근모드·사유·Step-up]
  E --> F[ContextSnapshot 발급]
  F --> G[TN 화면/API 재검증]
```

### 6.2 표준 배포

`PUBLISHED 표준 조회 → 대상 수임회사 선정 → 사전검증 → 배포일정/알림 → 테넌트 채택·매핑 → 실패/지연 재안내 → 완료 감사` 순으로 처리한다. 플랫폼 원본과 테넌트 채택본은 복제하지 않고 versionId로 참조한다.

### 6.3 기장 업무

`기장계약·모드 확인 → 담당배정 → BOOKKEEPING 컨텍스트 → 작성 → 독립 검토 → 마감/신고 승인 → 보고패키지 → 고객 Ack → 인계` 순이며 작성자와 검토자 SOD를 유지한다.

### 6.4 관계 종료·이관

OP-03에서 요청하고 PF-03 승인 워크플로를 따른다. `TRANSFER_IN_PROGRESS`에는 동시 쓰기를 차단하고 OP-08 인계 체크리스트와 OP-09 시크릿 회수를 완료한다.

| `ORG_TRANSFER_IN_PROGRESS` | 이관 중 쓰기 요청 |

## 9. 비기능·인수 기준

- OP 37개 화면은 PUBLISHED MenuVersion과 canonical ID로 노출된다.
- 모든 목록·집계·출력은 operatorOrgId를 서버에서 강제하고 교차조직 테스트를 통과한다.
- 관계·serviceScopes·배정·접근세션 변경은 진행 중 쓰기 요청의 커밋 직전 재검증된다.
- 중요 목록은 서버 페이징·정렬·필터·비동기 출력과 빈/오류/지연 상태를 지원한다.
- 권한 변경, 기장모드, Break-glass, 시크릿 위임, 키 회전은 Step-up·사유·SOD·감사를 통과한다.
- BOOKKEEPING 컨텍스트는 계약·관계·모드·배정 중 하나라도 없으면 발급되지 않는다.
- 표준/메뉴 원본 쓰기 API는 OP Role로 호출 시 거부된다.
- 배치 재처리는 멱등키·요청해시·DLQ 상태를 검증하고 중복 반영되지 않는다.
- 감사 해시체인과 세션/위임 만료·회수를 자동 테스트한다.
