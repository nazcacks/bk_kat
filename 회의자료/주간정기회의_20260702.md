# 주간회의 요약: KAT-BK meeting

**작성일:** 2026-07-02  
**회의명:** KAT-BK meeting  
**회의 상태:** 종료됨  
**회의 기록:** Teams 회의 전사 기반 요약  

---

## 1. 회의 목적

이번 회의는 BK 회계/세무 플랫폼의 **일반전표 입력(Journal Entry) 기능 설계**를 중심으로, 회계 시스템(WEHAGO, SAP, D365, K-System)의 장점을 반영한 차세대 회계 플랫폼의 UX/UI, 검증 로직, 외화 관리, 재무제표/세무 연계 기능을 논의하기 위해 진행되었다.

회의는 단순 화면 리뷰가 아니라, **전표 입력 → 계정과목 관리 → 외화 관리 → 결산 → STAT/TAX → 재무제표 주석/Disclosure**까지 연결되는 플랫폼 핵심 구조를 점검하는 성격이었다.

---

## 2. 주요 논의 사항

### 2.1 일반전표 입력 화면 설계

#### 입력 순서 유연화

현재 설계는 입력 순서가 고정되어 있어 사용자의 업무 방식에 따라 불편할 수 있다는 의견이 제시되었다. 사용자가 입력 순서를 자유롭게 구성할 수 있어야 한다는 방향으로 논의되었다.

예시 입력 순서:

- 거래처 → 계정과목 → 적요 → 금액
- 거래처 → 적요 → 계정과목 → 금액
- 계정과목 → 거래처 → 적요 → 금액
- 기타 사용자 정의 입력 순서

추가로 다음 기능이 필요하다는 의견이 있었다.

- 스마트 입력
- 자주 사용하는 계정
- 자동 분개 추천
- 과거 거래 및 계정 사용 이력 기반 추천

#### 첨부파일 및 증빙 연계

전표 입력 화면에서 첨부파일, OCR 증빙, 전자증빙을 직접 연결할 수 있어야 한다는 요구가 있었다. 전표 라인을 선택하면 해당 전표와 연결된 증빙이 즉시 조회되는 구조가 필요하다는 의견도 제시되었다.

주요 요구사항:

- 전표별 첨부파일 관리
- OCR 증빙 연계
- 전자증빙 연결
- 전표 선택 시 관련 증빙 자동 표시
- 증빙 첨부 여부 검증

#### 승인 Workflow

승인 기능은 업무 유형에 따라 선택적으로 적용해야 한다는 의견이 있었다.

논의 방향:

- 기장대행 서비스에서는 승인 절차가 필수적이지 않을 수 있음
- 일반 기업 판매 또는 GTM 버전에서는 승인 프로세스가 필요할 수 있음
- 회사별로 승인 기능 사용 여부를 선택할 수 있도록 설계
- 승인 상태, 미승인 상태, 로그 기록 관리 필요

### 2.2 외화(FX) 관리 기능

외화 관리는 회의에서 가장 중요하게 논의된 주제 중 하나였다.

#### 외화 입력 필드

외화 거래 입력 시 다음 항목이 필요하다는 의견이 있었다.

- 통화
- 환율
- 외화금액
- 원화 환산금액
- 발생 시 환율
- 회수 시 환율
- 환차손익
- 은행수수료

#### 외상매출금/외상매입금 추적

외화 거래는 단순 금액 입력이 아니라 Invoice 단위 추적과 연결되어야 한다는 논의가 있었다.

필요 기능:

- Invoice 단위 관리
- AR/AP 반제(Clearing)
- 선수금/선급금 자동 확인
- 입금과 인보이스 자동 매칭
- 부분 입금 처리
- 미수/미지급 잔액 추적

#### 환차손익 자동 계산

입금 또는 지급 시 발생 환율과 회수 환율의 차이를 분석하여 다음 분개가 자동 생성될 수 있어야 한다는 의견이 있었다.

- 외환차익
- 외환차손
- 은행수수료 비용
- 일부 입금에 따른 잔액 처리
- 차액/수수료 처리 옵션

#### 장기 연계 목표

외화 관리 기능은 단순 입력 기능에 그치지 않고 다음 업무와 연결되어야 한다는 방향이 제시되었다.

- 부가세 영세율 첨부서류
- 결산 시 외화평가
- 법인세 외화 관련 서식
- 외화 Aging 보고서
- 외화 채권/채무 추적

### 2.3 회계 통제 및 Validation 기능

#### 차대 일치 검증

차변 합계와 대변 합계가 일치하지 않으면 저장 또는 결재요청이 불가능해야 한다는 점이 중요하게 논의되었다.

처리 방향:

- 차변 합계 = 대변 합계 실시간 표시
- 차액 발생 시 우측 패널에 표시
- 불일치 시 저장 또는 결재요청 차단
- 차액 전표보기 또는 차액 라인 강조 기능 검토

#### 보조부 필수 입력

보조부 필수 입력은 계정별, 회사별 설정에 따라 다르게 적용해야 한다는 의견이 있었다.

예시:

- AR/AP 계정: 거래처 필수
- 자금 계정: 자금관리 항목 필수
- 차량 관련 계정: 차량 정보 필수
- 프로젝트 비용 계정: 프로젝트 필수

필수 보조항목은 계정과목 입력 시 자동 추천되거나 자동 노출되는 방식이 적절하다는 논의가 있었다.

#### 적요 미입력 검증

적요 누락은 감사 및 세무조사 시 이슈가 될 수 있으므로 단순 경고 수준으로 관리할지, 필수 입력으로 관리할지 검토가 필요하다는 의견이 있었다.

제안 기능:

- 표준 적요 템플릿
- 과거 적요 추천
- 계정/거래처/전표유형별 적요 자동 제안
- 불명확 적요 경고
- 적요와 증빙 제목/공급자명 매칭
- 적요 전문 검색

#### 거래처 검증

자산 및 부채 계정에는 거래처 입력이 필수여야 한다는 의견이 강하게 제시되었다.

주요 검증 항목:

- 미등록 거래처 경고 또는 차단
- 자산/부채 계정 거래처 필수
- 휴폐업/사용중지 거래처 차단
- 사업자번호 중복 경고
- AR 계정에 공급업체 입력 방지
- AP 계정에 고객 입력 방지
- 특정 거래처는 특정 계정만 허용

#### 중복 전표 검출

동일하거나 유사한 전표 입력을 방지하기 위한 중복 검출 기능이 필요하다는 의견이 있었다.

중복 판단 기준:

- 일자
- 전표유형
- 계정
- 거래처
- 금액
- 적요 유사도
- 동일 카드 승인건 여부
- 동일 Invoice 여부

중복 가능성이 있을 경우 경고 후 사용자가 확인하도록 하는 방식이 논의되었다.

### 2.4 날짜 구조 개선

기존 전표 입력 구조에서 날짜 개념이 단순하므로, 회계/세무/증빙 관점에서 날짜를 구분해야 한다는 논의가 있었다.

검토된 날짜 항목:

- 전표일자: 회계 반영 기준
- 증빙일자: 증빙 발생 기준
- 기표일자 또는 작성일자: 실제 입력 또는 POSTED 처리 기준
- 세무귀속일: 세무신고 기준

주요 논의 내용:

- 원칙적으로 전표일자와 증빙일자는 일치하는 것이 바람직함
- 해외 ERP 또는 외부 업로드 데이터에서는 전표일자와 증빙일자가 불일치하는 사례가 있음
- VAT 신고 기준과 회계 반영 기준의 차이를 검토해야 함
- 전표일자와 증빙일자가 불일치할 경우 경고 또는 검토 요청 필요
- 마감기간에 해당하는 전표일자는 차단 필요
- 미래일자 입력은 차단 필요

### 2.5 계정과목(COA) 관리

강수님은 계정과목 관리 구조에 대해 설명하였다. 핵심 방향은 회사별 계정, IFRS, 법인세 표준계정, 국가별 GAAP 계정을 공통 카탈로그와 매핑 구조로 관리하는 것이다.

#### 주요 설계 방향

- 회사계정 관리
- IFRS 계정 관리
- 법인세 표준계정 관리
- 국가별 GAAP 계정 관리
- 공통 템플릿 기반 회사별 COA 생성
- 엑셀 업로드를 통한 초기화 가능
- 회사별로 기본 계정 체계를 복제 및 수정 가능

#### 코드 길이 유연화

고객사마다 계정 코드 길이가 다를 수 있으므로 3자리, 5자리, 8자리, 9자리 등 가변 코드 체계 지원이 필요하다는 의견이 있었다.

#### 계정별 통제 속성

계정과목별로 다음과 같은 속성을 관리해야 한다는 논의가 있었다.

- 정상잔액
- 거래처 입력 필수 여부
- 외화금액 등록 여부
- 관리항목 노출/필수 여부
- 사업장/부서/프로젝트 필수 여부
- 증빙 첨부 필요 여부
- 자동결산 대상 여부
- 자금관리 연동 여부
- 부가세 유형
- 결산대체 관계

- Stat Conversion 자동화 가능성

## 3. 결정 사항

1. **일반전표 입력 기능을 최우선 개발 대상으로 진행한다.**
2. **계정과목 관리와 전표 입력 기능이 핵심 선행 과제라는 점에 동의하였다.**
3. **승인 Workflow는 회사별 옵션 기능으로 설계하는 방향을 검토한다.**
4. **자산/부채 계정의 거래처 필수 입력 방향에 대체로 동의하였다.**
5. **외화 관리 기능은 핵심 기능으로 유지하며, AR/AP 반제 및 환차손익 계산과 연결한다.**
6. **날짜 구조는 최소한 전표일자, 증빙일자, 기표일자를 분리하는 방향으로 검토한다.**
7. **STAT/TAX/IFRS 연계 기능은 장기적으로 중요한 차별화 기능으로 인식하였다.**
8. **주석/Disclosure 자동화 기능은 향후 플랫폼 경쟁력 확보에 중요한 기능으로 검토한다.**
9. **VAT 및 카드 매입세액 분석 기능은 핵심 전표 입력 이후 단계적으로 검토한다.**
10. **기능 범위가 매우 넓으므로 우선 전표 입력과 계정과목 관리부터 완성한 뒤 추가 기능을 순차적으로 반영한다.**

## 6. 회의 총평

이번 회의는 BK 회계/세무 플랫폼의 장기적인 제품 방향을 구체화한 중요한 논의였다. 특히 **일반전표 입력과 계정과목 관리가 전체 플랫폼의 기반 기능**이라는 점이 명확해졌으며, 이를 바탕으로 외화 관리, AR/AP 반제, 세무조정, STAT 변환, 재무제표 주석 자동화까지 확장할 수 있다는 방향성이 확인되었다.

다만 논의된 기능 범위가 매우 넓기 때문에, 단기적으로는 **전표 입력과 계정과목 관리의 MVP 완성**에 집중하고, 외화/STAT/TAX/Disclosure/VAT 분석 기능은 단계적으로 추가하는 접근이 적절하다.
