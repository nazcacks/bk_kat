문서 안내

이 문서는 system/ 아래의 실행 코드와 설정을 직접 추적해 작성했습니다. 상위 설계서의 모든 미래 기능을 설명하는 문서가 아니라, 현재 동작하는 프레임워크와 명시된 후속 과제를 구분하는 것이 목적입니다.

한 줄 결론

현재 시스템은 운영 콘솔과 공통 보안·메뉴·CRUD 기반이 구현된 회계·세무 SaaS 골격입니다. 실제 회계 업무 도메인은 대부분 메뉴와 Placeholder까지 준비된 상태이며, 전표·결산·세무 로직은 후속 페이즈에서 채워야 합니다.

01 · System overview

시스템 개요

3개 채널(OP/TN/CO)의 메뉴 카탈로그를 중심으로, 운영 콘솔과 이용회사 업무 화면을 하나의 React 애플리케이션에서 제공하는 멀티테넌트 SaaS 지향 구조입니다.

108TypeScript/TSX 소스 파일
10,606소스 코드 라인
292시드 메뉴 노드
33HTTP 엔드포인트
13구현 컴포넌트 라우트

측정 범위: system/backend/src 50개·2,936줄, system/frontend/src 58개·7,670줄. 공백/주석을 포함한 물리 라인 기준입니다.

채널 모델

OP · 운영콘솔

시스템 운영자용. 테넌트, 사용자, 인증, 권한, 메뉴, 공통코드, 배치, 로그, 개인정보보호 화면을 포함합니다.

TN · 업무화면

이용회사 회계·세무 업무용. 대시보드는 구현됐고 기초정보, 전표, 장부, 결산, 부가세 등 대부분은 Placeholder입니다.

CO · 공통

로그인, 공통 UI, API 계약, 오류·관측성, 산출물 추적 등 채널 공통 기반을 의미합니다.

02 · Architecture

전체 아키텍처

개발 환경에서는 Vite가 /api를 NestJS로 프록시하고, NestJS가 TypeORM을 통해 PostgreSQL에 접근합니다. 프론트 조회 API는 서버 장애 시 mock으로 폴백하도록 설계되어 있습니다.

Browser · :5173
  • React Router SPA
  • Zustand + localStorage
  • Axios API client
  • 조회 mock fallback
→
NestJS API · :3000/api
  • Request context
  • JWT guard / validation
  • Controller → Service
  • Audit + response envelope
→
PostgreSQL · :5432
  • TypeORM repositories
  • 업무/보안 테이블
  • JSONB 운영 리소스
  • 시작 시 빈 테이블 seed
메뉴 이중 소스
사이드바는 API 메뉴 트리를 읽지만 라우트 목록은 빌드 시점의 mockMenuTree로 생성됩니다.
감사 분기
성공한 변경 요청은 감사 인터셉터가 보안 모듈로 전달하고, 로그인은 별도 LoginHistory로 기록합니다.

HTTP 요청 생명주기

요청 컨텍스트 생성X-Request-Id를 수용하거나 UUID를 발급하고 응답 헤더로 되돌립니다. X-Tenant-Id도 읽지만 현재 권한 검증에는 사용하지 않습니다.
전역 JWT 가드@Public() 여부를 확인하고 Bearer 토큰을 검증합니다. 개발 설정 AUTH_BYPASS=true면 토큰 없이 개발 사용자로 통과합니다.
전역 ValidationPipeDTO가 정의된 요청은 변환·화이트리스트·class-validator 검증을 받습니다. 현재 변경 API 다수는 전용 DTO가 없어 검증 효과가 제한적입니다.
Controller → Service컨트롤러가 파라미터를 받고 서비스가 업무 검증과 TypeORM Repository 호출을 수행합니다.
성공 변경 감사POST/PUT/PATCH/DELETE 또는 @Audit() 호출은 성공 후 요청 본문을 정제해 해시체인 감사로그로 비동기 기록합니다.
표준 응답/오류 변환성공은 {success,data,error,meta} 봉투, 실패는 전역 예외 필터가 동일한 오류 봉투로 변환합니다.
Axios 응답 해제프론트 응답 인터셉터가 성공 봉투의 data만 반환합니다. 401이면 인증 스토어를 비우고 로그인 화면으로 이동합니다.

의존 방향

백엔드

Controller → Service → Repository(Entity)가 기본 방향이며, 모든 모듈은 전역 SecurityModule의 SecurityService를 사용할 수 있습니다. 공통 계층이 도메인 모듈을 직접 참조하는 예외는 감사 인터셉터가 SecurityService를 호출하는 부분입니다.

프론트엔드

Page → api/* → api/client, Page → components, Layout → stores 구조입니다. 임시 운영 리소스 화면은 Page → useResourceCrud → api/resources 경로를 공유합니다.

06 · API reference

07 · Data model

데이터 모델

현재 스키마는 운영 기반 엔티티 중심입니다. 회계 전표·계정·원장 등 핵심 업무 테이블은 아직 존재하지 않습니다. 엔티티 간 연결은 외래키보다 코드 값으로 느슨하게 연결된 부분이 많습니다.

09 · Core flows

핵심 처리 흐름

프레임워크를 이해할 때 파일별 설명보다 실제 사용자 동작이 어느 계층을 통과하는지 보는 편이 빠릅니다.

1. 로그인

LoginPage 제출api/auth.login()이 POST /auth/login을 호출합니다.
사용자 조회/개발 생성AuthService가 loginId를 조회합니다. bypass 상태에서 미존재 사용자는 TENANT_ADMIN으로 자동 생성됩니다.
검증과 이력운영 모드 의도상 bcrypt를 검증하고, 성공·실패·BYPASS를 login_history에 기록합니다.
JWT 발급/저장사용자 식별·그룹·tenantId·roles를 payload로 발급하고 authStore/localStorage에 저장합니다.
대시보드 진입Router가 /dashboard로 이동합니다. 이후 API 401 발생 시 저장 상태를 제거합니다.

2. 메뉴 로딩과 화면 선택

  1. 앱 번들 평가 시 mockMenuTree에서 전체 MENU 경로의 React Router route를 생성합니다.
  2. MainLayout 마운트 후 menuStore.loadMenus()가 /menus/tree를 호출합니다.
  3. 성공하면 API 트리, 실패하면 mock 트리를 Sidebar에 표시합니다.
  4. 현재 경로의 조상 GROUP을 자동으로 펼치고 채널(OP/TN/CO)을 맞춥니다.
  5. 구현 경로는 실제 Page, 미구현 경로는 메뉴명과 screenId를 보여주는 Placeholder로 렌더링합니다.

4. 개인정보 평문 조회

  1. 기본 사용자 목록은 email/phone을 마스킹해 반환합니다.
  2. 평문이 필요하면 GET /users/:id/plain?reason=...을 호출합니다.
  3. 사유가 없거나 2자 미만이면 PRIVACY_REASON_REQUIRED로 거부합니다.
  4. 호출자·대상·항목·목적·IP·requestId를 personal_data_access_log에 기록합니다.
  5. 그 후 평문 데이터를 반환합니다. 현재 별도의 민감정보 grant/Step-up 검증은 없습니다.

5. 변경 감사

  1. 성공한 변경 요청에서 actor, tenant, HTTP 정보, 요청 본문을 수집합니다.
  2. password/secret/token/rrn/residentNo 키는 ***로 치환합니다.
  3. 마지막 감사행의 chainHash를 prevHash로 읽습니다.
  4. 일부 핵심 필드 JSON을 SHA-256하여 새 chainHash를 저장합니다.
  5. 기록 실패는 업무 응답을 실패시키지 않고 서버 로그만 남깁니다.

A. 정식 백엔드 모듈 추가

  1. modules/<domain>/에 entity, dto, service, controller, module을 만듭니다.
  2. 입력은 Partial<Entity> 대신 class-validator가 적용된 전용 DTO로 정의합니다.
  3. 테넌트 데이터는 JWT actor의 tenantId를 서버에서 주입하고 쿼리 조건에 강제합니다.
  4. 변경 핸들러에 구체적인 @Audit(resource, action)을 붙입니다.
  5. AppModule imports에 모듈을 등록합니다.
  6. 운영을 고려해 migration과 테스트를 함께 추가합니다.
DTO example
export class CreateCompanyDto {
  @IsString()
  @IsNotEmpty()
  companyCode: string;

  @IsString()
  @Length(1, 100)
  name: string;

  @IsOptional()
  @IsIn(['ACTIVE', 'SUSPENDED'])
  status?: string;
}
Controller pattern
@Post()
@Audit('COMPANY', 'CREATE')
create(
  @Body() dto: CreateCompanyDto,
  @CurrentUser() actor: AuthUser,
) {
  return this.companiesService.create(dto, actor.tenantId);
}

  • TN 업무 메뉴 200여 화면은 Placeholder이며 실제 전표·장부·결산·세무 엔티티/서비스가 없습니다.

14 · Troubleshooting

문제 해결

개발 환경에서 자주 발생할 수 있는 증상을 현재 코드의 동작 특성과 연결해 정리했습니다.

백엔드가 PostgreSQL에 연결되지 않습니다
  1. docker compose ps에서 bk-postgres가 healthy인지 확인합니다.
  2. 5432 포트 충돌과 .env의 DB_HOST/PORT/USERNAME/DATABASE를 확인합니다.
  3. docker compose logs postgres로 인증·볼륨 오류를 확인합니다.
  4. 앱은 연결을 3회, 2초 간격으로 재시도합니다.
백엔드가 꺼져 있는데 화면에는 데이터가 보입니다

조회 API가 mock으로 폴백하기 때문입니다. 브라우저 콘솔의 ... 실패 — mock으로 폴백 경고를 확인하십시오. 등록·수정·삭제는 fallback이 없어 실패하는 것이 정상입니다.

새 메뉴가 사이드바에는 보이지만 화면 라우트가 이상합니다

사이드바는 API 메뉴, 라우터는 mockMenuTree를 사용합니다. menu.seed.ts, mock/menus.ts, router.tsx의 경로 등록을 모두 확인합니다.

시드 파일을 바꿨는데 데이터가 갱신되지 않습니다

SeedService는 테이블에 한 건이라도 있으면 건너뜁니다. 개발 DB 볼륨을 초기화하면 전체 데이터가 사라지므로 주의하고, 보존해야 할 데이터가 있으면 migration/관리 스크립트를 작성합니다.

AUTH_BYPASS=false에서 로그인이 되지 않습니다

현재 password_hash가 select:false이며 일반 조회가 해당 컬럼을 명시적으로 선택하지 않습니다. 인증 전용 query builder로 hash를 선택하도록 보완한 뒤 bcrypt 검증 테스트를 수행해야 합니다.

한글이 깨져 보입니다

소스와 이 문서는 UTF-8입니다. PowerShell에서 읽을 때 Get-Content -Encoding UTF8을 사용하고, 편집기 저장 인코딩과 HTML의 <meta charset="utf-8">를 확인합니다.

운영에서 DB 컬럼이 자동 변경됩니다

DB_SYNCHRONIZE=true 때문입니다. 운영에서는 false로 고정하고, 배포 전에 명시적 TypeORM migration을 생성·검토·백업·적용해야 합니다.
